# V2 左侧菜单与用户管理页面技术方案

## 1. 背景和目标

当前前端左侧导航在 `AI_RAG_Agent_Frontend/src/app/navigation.ts` 中写死，`AppShell.vue`
通过 `portalPages` 渲染菜单，并通过 `activePage` 在 `App.vue` 中切换页面。

本次目标：

1. 左侧导航改成菜单样式，并由后端菜单表驱动。
2. 新增“系统管理 / 用户管理 / 角色管理”菜单。
3. 用户管理页面支持系统用户增删改查。
4. 角色管理页面支持角色增删改查和菜单授权。
5. 本次新增表主键统一使用雪花算法 ID，数据库类型使用 `BIGINT`。
6. 保持现有 V2 入口：后端统一走 `/api/v2`，前端继续使用当前 `AppShell + activePage` 模式，先不引入 Vue Router。

## 2. 当前代码现状

### 后端

- 应用入口：`api/main.py`
- V2 路由总入口：`app_v2/api/router.py`
- 认证路由：`app_v2/api/routes/auth.py`
- 当前用户表实体：`domain/entities.py::SystemUserEntity`
- 当前用户表：`system_users`
- 当前登录能力：`/api/v2/auth/login`、`/api/v2/auth/refresh`、`/api/v2/auth/me`、`/api/v2/auth/logout`

已有 `system_users` 表字段：

| 字段 | 说明 |
| --- | --- |
| `user_id` | 用户唯一编号 |
| `username` | 登录账号 |
| `display_name` | 展示名称 |
| `password_hash` | 密码哈希 |
| `role` | 用户角色，当前主要是 `admin` / `user` |
| `status` | 用户状态，`active` 才允许登录 |
| `last_login_at` | 最后登录时间 |

### 前端

- 左侧壳组件：`src/app/AppShell.vue`

- 类型定义：`src/shared/api/types.ts`

当前页面没有使用 Vue Router，而是通过 `activePage` 控制：


## 3. 总体设计

### 3.1 分层

后端新增系统模块：

```text
app_v2/
  api/routes/system.py                  # 菜单、用户管理、角色管理接口
  application/system_service.py          # 菜单树、用户管理、角色管理应用服务
  infrastructure/repositories/system_repository.py
  domain/schemas.py                      # 也可以拆到 system_schemas.py，二选一
domain/entities.py                       # 新增 SystemRoleEntity、SystemMenuEntity、SystemRoleMenuEntity
docs/V2左侧菜单与用户管理页面设计/06-mysql迁移_V2左侧菜单与用户管理.sql
```

前端新增系统模块：

```text
src/shared/api/system.ts                 # 菜单、用户管理、角色管理 API
src/app/navigation.ts                    # 保留 pageKey 到组件的类型映射
src/app/AppShell.vue                     # 从接口菜单渲染左侧菜单树
src/App.vue                              # 增加 userManagement、roleManagement 页面切换
src/features/system/UserManagementPage.vue
src/features/system/RoleManagementPage.vue
```

### 3.2 菜单模型

菜单表只负责“页面入口”和“展示结构”，不直接承载复杂权限逻辑。

本次设计：

- `system_roles` 保存角色。
- `system_menus` 保存菜单树。
- `system_role_menus` 保存角色和菜单关系，使用 `role_id + menu_id` 建立关联。
- `system_users.role` 保存当前用户所属角色编码，对应 `system_roles.role_code`。
- 登录后前端调用 `/api/v2/system/menus/me` 获取当前用户可见菜单。

这样可以直接支持：

- 管理员看到全部菜单、用户管理和角色管理。
- 普通用户只看到业务页面。
- 后续新增角色时，在角色管理页面勾选菜单即可。

### 3.3 主键和 ID 传输规则

从本需求开始，新增表主键统一按雪花算法生成：

- 数据库主键字段使用 `BIGINT`，例如 `system_roles.role_id`、`system_menus.menu_id`、`system_role_menus.role_menu_id`。
- 新增关系字段也使用目标表的雪花 ID，例如 `system_role_menus.role_id`、`system_role_menus.menu_id`。
- 历史表和历史数据本期不强制迁移，例如 `system_users.user_id` 字段类型和旧值保持现状。
- 用户管理中新建 `system_users` 数据时，`user_id` 使用字符串化雪花 ID 写入现有 `VARCHAR` 字段。
- 后端 ORM 实体可使用 Python `int` 承接数据库 `BIGINT`。
- API 返回给前端时，雪花 ID 必须转成字符串，避免 JavaScript `number` 精度丢失。

### 3.4 用户管理模型

用户管理直接复用 `system_users`，不新增用户主表。

用户管理接口需要注意：

- 列表和详情不能返回 `password_hash`。
- 新增用户必须设置初始密码。
- 修改用户时密码可选，不传则不改密码。
- 用户角色必须从启用的 `system_roles` 中选择，业务上继续保存 `role_code`。
- 删除建议第一期做逻辑禁用或状态改为 `disabled`，不做物理删除。
- 禁止管理员禁用或删除自己，避免把当前账号锁死。

### 3.5 角色管理模型

角色管理页面管理 `system_roles` 和 `system_role_menus`。

角色管理接口需要注意：

- `role_code` 全局唯一。
- 内置角色 `admin`、`user` 不允许删除，只允许修改名称、说明和菜单授权。
- 禁用角色前需要校验是否仍有启用用户使用该角色。
- 角色菜单授权以菜单 ID 列表整体覆盖，后端负责保存到 `system_role_menus`。
- 管理员角色默认拥有全部菜单，不建议移除“角色管理”和“用户管理”，避免锁死后台。

## 4. 业务规则

### 菜单规则

1. 菜单分两类：
   - `directory`：目录，只做分组，例如“系统管理”。
   - `page`：页面菜单，可以点击切换页面，例如“用户管理”。
2. `visible = 1` 才返回给前端。
3. `status = active` 才可用。
4. 同级按 `sort_order ASC, created_at ASC` 排序。
5. 前端点击 `page` 类型菜单时，通过 `page_key` 切换本地页面。
6. 当前不引入 Vue Router，所以 `route_path` 先作为兼容字段保留。

### 用户规则

1. `username` 全局唯一，长度 1 到 128。
2. `display_name` 必填，长度 1 到 128。
3. `role` 必须来自启用的 `system_roles.role_code`。
4. `status` 第一期限定为 `active`、`disabled`。
5. 创建用户时密码必填，长度建议 6 到 128。
6. 修改用户时密码可选；为空或不传表示不修改。
7. 禁止禁用当前登录用户。
8. 禁止删除当前登录用户。
9. 删除用户第一期建议实现为 `status = disabled`，保留历史登录和业务关联。

## 5. 权限设计

权限按角色菜单关系控制：

| 角色 | 菜单权限 | 用户管理权限 |
| --- | --- | --- |
| `admin` | 全部菜单 | 用户管理、角色管理增删改查 |
| `user` | 首页、智能客服、销售陪练、问答考试 | 无系统管理入口，无系统管理接口权限 |

后端系统管理接口第一期统一要求当前用户 `role = admin`。

如果普通用户直接调用接口，返回：

```json
{
  "detail": "无权限访问系统管理"
}
```

HTTP 状态码使用 `403`。

## 6. 页面结构

左侧菜单建议结构：

```text
首页
智能客服
销售陪练
问答考试
系统管理
  用户管理
  角色管理
```

用户管理页面建议采用后台管理风格：

- 顶部工具栏：关键词搜索、状态筛选、角色筛选、新增用户按钮。
- 主体表格：账号、展示名称、角色、状态、最后登录时间、更新时间、操作。
- 操作：编辑、重置密码、禁用/启用。

角色管理页面建议采用左右结构：

- 左侧角色列表：角色编码、角色名称、状态、是否内置、操作。
- 右侧角色表单：角色基础信息。
- 右侧菜单授权树：勾选该角色可见菜单。
- 操作：新增角色、编辑角色、禁用/启用角色、保存菜单授权。
- 新增/编辑使用弹窗表单。

## 7. 兼容和取舍

1. 暂不引入 Vue Router。
   - 原因：当前页面切换模式稳定，先改菜单来源即可。
   - 后续如果页面增多，再把 `page_key` 映射迁移到路由表。
2. 用户删除采用禁用。
   - 原因：`conversations`、`exam_sessions` 等表已经有 `user_id`，物理删除会影响审计。
3. 权限先做到菜单级。
   - 原因：当前需求是左侧菜单和用户/角色页面，不需要马上做按钮级权限。
   - 后续如果需要按钮权限，可新增 `system_permissions` 和 `system_role_permissions`。

## 8. 实施顺序

1. 建表：`system_roles`、`system_menus`、`system_role_menus`，初始化角色和菜单数据。
2. 后端新增系统菜单接口：`GET /api/v2/system/menus/me`。
3. 后端新增用户管理接口：列表、创建、修改、禁用、重置密码。
4. 后端新增角色管理接口：列表、创建、修改、禁用、菜单授权。
5. 前端新增系统 API 和类型。
6. `AppShell.vue` 改为接口菜单渲染。
7. `App.vue` 增加 `userManagement`、`roleManagement` 页面。
8. 新增 `UserManagementPage.vue`、`RoleManagementPage.vue`。
9. 补充后端和前端测试。

## 9. 风险点

| 风险 | 处理方式 |
| --- | --- |
| 菜单接口失败导致页面空白 | 前端保留静态菜单兜底或展示错误态 |
| 当前用户误禁用自己 | 后端明确拦截 |
| 普通用户绕过前端直接调系统管理接口 | 后端按 `role` 做强校验 |
| 角色被禁用导致用户无法登录或菜单为空 | 禁用角色前校验是否仍有启用用户使用 |
| 管理员角色误删关键菜单权限 | 内置 `admin` 角色禁止删除，并限制移除关键后台菜单 |
| 菜单 `page_key` 和前端组件不一致 | 前端维护白名单映射，未知菜单不渲染或置灰 |
| 删除用户影响历史数据 | 第一阶段只禁用，不物理删除 |
