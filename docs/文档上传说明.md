# 文档上传说明

本文档说明当前项目中文档上传的完整规则、接口、前端操作、入库结果和排查方式。

当前项目有两套上传链路，不能混用：

| 上传类型 | 前端入口 | 后端接口前缀 | 默认 Qdrant Collection | MySQL 表 | 主要用途 |
| --- | --- | --- | --- | --- | --- |
| 普通知识库上传 | 首页 / 知识库管理 | `/knowledge` | `agent`，也可前端选择或输入其他 collection | `documents` | 智能客服 RAG、考试题源、普通知识问答 |
| 销售训练资料上传 | 销售训练页 / 训练知识 | `/training/knowledge` | `sales_training_cases`，代码固定 | `documents` + `training_knowledge_batches` | AI 客户生成、训练对话、评分复盘 |

## 1. 上传前准备

### 1.1 必须启动的服务

上传入库依赖后端、Qdrant、Embedding 模型。

| 依赖 | 作用 | 未启动时表现 |
| --- | --- | --- |
| FastAPI 后端 | 接收上传请求、解析文件、写 MySQL、写 Qdrant | 前端接口报错或无法连接 |
| Qdrant | 保存向量点和 metadata | 入库失败、健康检查 qdrant 不可用 |
| Embedding 模型配置 | 把文本切片转换成向量 | `add_documents` 或检索阶段失败 |
| MySQL `ai_rag_agent` | 保存文件、批次、切片、会话等业务元数据 | 查询文件列表、训练资料列表失败 |

本地 Qdrant 推荐命令：

```powershell
docker run -d --name qdrant -p 6333:6333 -p 6334:6334 qdrant/qdrant
```

当前 `config/qdrant.yml` 中默认使用 HTTP：

```yaml
url: http://localhost:6333
grpc_port: 6334
prefer_grpc: false
```

也就是说项目默认走 `6333`，不是 `6334`。

### 1.2 允许上传的文件类型

当前配置和代码支持：

| 上传链路 | 支持后缀 | 来源 |
| --- | --- | --- |
| 普通知识库 | `txt`、`pdf`、`docx` | `config/qdrant.yml` 的 `allow_knowledge_file_type` |
| 销售训练资料 | `txt`、`pdf`、`docx` | `training/services/sales_training_service.py` 的 `ALLOWED_TRAINING_FILE_TYPES` |

注意：

| 文件类型 | 入库支持 | 已入库文件预览支持 | 说明 |
| --- | --- | --- | --- |
| `txt` | 支持 | 支持 | 按 UTF-8 文本读取 |
| `pdf` | 支持 | 支持 | 抽取 PDF 文本；图片型扫描 PDF 可能没有有效文本 |
| `docx` | 支持 | 普通知识库预览暂未支持；训练资料预览支持 | 可以解析 Word 段落并入库，但 `/knowledge/files/{document_id}/preview` 当前只处理 `txt/pdf` |

### 1.3 文件内容要求

上传文件必须包含可抽取文本。以下文件不适合直接上传：

| 文件情况 | 风险 | 建议 |
| --- | --- | --- |
| 扫描版 PDF，内容是图片 | 文本抽取为空，无法切片 | 先 OCR 成可复制文本 |
| 大量表格截图 | 解析不稳定 | 转成清晰 Markdown/TXT/DOCX |
| 文件为空 | 后端直接拒绝 | 补充正文后再上传 |
| 文件名包含路径 | 后端会清理路径，只保留文件名 | 正常选择本地文件即可 |
| 文件内容重复 | 后端按 MD5 去重 | 如确实要重复入不同 collection，普通知识库可选择不同 collection；训练资料目前复用已发布批次 |

## 2. 普通知识库上传

普通知识库用于智能客服聊天、知识问答和考试题源。该链路会先预览，再由用户确认后正式入库。

### 2.1 前端操作流程

1. 打开首页。
2. 进入“知识库管理”。
3. 点击上传按钮选择文件。
4. 后端先执行上传预览，前端弹出确认窗口。
5. 在确认窗口中检查：
   - 文件名；
   - 文件大小；
   - 是否重复；
   - 系统识别的文档结构；
   - 系统建议的切分策略；
   - 抽样文本。
6. 如需更稳妥，可点击“模型推荐切分方式”。
7. 选择或输入目标 collection。
8. 点击确认入库。
9. 回到知识库文件列表，确认状态为 `indexed`，并检查 `chunk_count`。

### 2.3 文档结构类型

当前支持的 `document_type`：

| 值 | 中文含义 | 适合文件 |
| --- | --- | --- |
| `qa` | 问答型文档 | FAQ、100问、题库问答 |
| `numbered` | 编号条目型文档 | 选购指南 1-200 条、规则清单 |
| `text` | 普通文本型文档 | 普通说明书、文章、无稳定编号结构的资料 |

### 2.4 切分策略

当前支持的 `split_strategy`：

| 值 | 中文含义 | 切分方式 | 适合文件 |
| --- | --- | --- | --- |
| `numbered_qa` | 编号问答切分 | 按 `1. 问题`、`2. 问题` 等编号块切出“问题 + 答案” | TXT/DOCX 中清晰的 FAQ、100问 |
| `outline_qa` | PDF 目录问答切分 | 根据 PDF 书签目录定位问题，再抽取答案 | 带规范书签目录的 PDF 问答文档 |
| `numbered_segments` | 编号条目切分 | 按编号条目切分成独立 segment | 指南类、条目类资料 |
| `recursive` | 递归文本切分 | 按段落、换行、标点、空格逐级切分 | 普通文档兜底策略 |

建议：

| 文件形态 | 推荐配置 |
| --- | --- |
| “扫地机器人100问”这种编号问答 TXT/DOCX | `document_type=qa`，`split_strategy=numbered_qa` |
| 带 PDF 书签目录，目录下每项都是问题 | `document_type=qa`，`split_strategy=outline_qa` |
| “选购指南200条”这种编号知识点 | `document_type=numbered`，`split_strategy=numbered_segments` |
| 无明显结构的说明文档 | `document_type=text`，`split_strategy=recursive` |

## 3. 销售训练资料上传

销售训练资料用于生成 AI 客户、陪练对话、训练评分。它和普通聊天知识库分开保存：

- 待审核切片写入临时 Qdrant collection：`sales_training_cases_staging`。
- 人工确认后复制到正式 Qdrant collection：`sales_training_cases`。

当前项目一期先做单用户功能闭环，不区分管理端和员工端，也不在上传阶段配置权限、画像、任务难度等信息。
上传阶段只做“待审核入库”：保存原文件、重复校验、解析、切片、质量评估，并把切片正文写入临时向量库。
MySQL `documents` 统一保存文件基础信息，`training_knowledge_batches` 保存批次状态和质量报告，不再保存新切片正文；确认发布后才进入正式训练向量库。
画像类型、训练任务类型、行业、难度、训练轮数、评分规则都属于后续创建训练任务或生成训练设置时使用的信息，不放在上传表单里。

### 3.1 前端操作流程

1. 打开“销售训练”页面。
2. 在“训练知识”区域选择文件。
3. 文件选择框当前允许 `.docx,.pdf,.txt`。
4. 点击上传。
5. 后端默认按 `lms_case` 来源类型处理。
6. 上传成功后，页面会展示：
   - 原文件名；
   - 入库状态；
   - 切片数量；
   - 向量点数量；
   - 是否命中重复文件；
   - 切片质量报告；
   - 切片正文和结构化 metadata。
7. 如果切片质量不满意，可以点击“LLM 重新切分”，重新生成预览切片。
8. 人工检查通过后，点击“确认发布到训练库”，临时向量点会复制到 `sales_training_cases`。
9. 可以在历史批次里预览原文件、查看切片、查看版本链、回滚版本、删除批次。

一期前端不展示这些字段：

| 字段 | 一期不展示原因 |
| --- | --- |
| 画像类型 | 属于训练任务上下文，不是文件属性 |
| 训练任务类型 | 一期固定开放式销售训练，不需要上传时选择 |
| 行业 | 可以后续作为检索标签，一期先不增加上传复杂度 |
| 难度 | 由训练设置阶段的 LLM 动态生成，不属于上传资料 |
| 上传人 | 当前没有用户体系，先不展示 |
| 可见性 | 当前没有管理端/员工端权限隔离，一期先不做 |

### 3.3 `source_type` 说明

`source_type` 已做成系统字典，字典编码为 `training_source_type`。

当前前端销售训练页只开放已经完整实现的类型：

| 字典编码 | 字典项编码 | 字典项名称 | 当前处理策略 | 适用文件 |
| --- | --- | --- | --- | --- |
| `training_source_type` | `lms_case` | LMS 销售训练案例 | 使用 `LmsCaseIngestStrategy` 结构化拆分 | LMS 案例、销售训练任务、话术案例 |

`lms_case` 字典项 metadata：

| metadata 字段 | 当前值 | 中文含义 |
| --- | --- | --- |
| `default` | `true` | 作为训练资料来源类型默认值 |
| `implemented` | `true` | 已完成专门处理逻辑 |
| `strategy` | `LmsCaseIngestStrategy` | 后端使用的切片策略类 |
| `collection_name` | `sales_training_cases` | 发布后的正式训练向量库 collection |

产品资料、FAQ、竞品资料、成功案例、术语表、人工录入资料等类型暂不作为前端可选项开放。后续如果要支持，建议先新增对应字典项，再补充专门 `KnowledgeIngestStrategy`，避免用户误以为已经完整实现。

### 3.4 `lms_case` 推荐文档结构

为了让系统准确拆分训练资料，建议 LMS 案例文件按下面结构写：

```text
一、客户案例标题

客户案例：
客户背景、行业、角色、合作阶段等。

任务要求：
学员需要完成什么沟通任务。

匹配答案：
优秀话术、标准回答、参考打法。

隐性心理：
客户真实顾虑、未明说动机、底层担忧。

评分标准：
命中点、扣分点、能力维度。

二、客户案例标题
...
```

标题关键词识别规则：

| 切片类型 `case_part` | 识别关键词 | 一期用途 |
| --- | --- | --- |
| `case_profile` | `客户案例`、`企业`、`老板身份`、`合作行业`、`客户阶段` | 生成客户角色和对话背景 |
| `task_requirement` | `任务要求` | 生成训练目标 |
| `standard_answer` | `匹配答案`、`话术案例`、`话术原文`、`谈单话术`、`参考答案` | 训练评分和回答建议 |
| `hidden_psychology` | `隐性心理`、`底层顾虑`、`底层需求`、`客户底层心理`、`核心显性卡点`、`隐性痛点` | 生成 AI 客户的追问、顾虑和异议 |
| `scoring_rubric` | `命中点`、`扣分点`、`评分标准`、`能力维度` | 训练结束评分 |

如果文档没有识别出结构，系统会兜底生成一个 `case_profile` 切片，但训练效果会明显变差。

### 3.5 可见性说明

当前项目一期没有管理端和员工端权限区分，因此上传页面不展示可见性配置。

后端切片里仍可以保留 `visibility` 字段，原因是它能帮助 LLM 判断资料用途：

| 值 | 一期含义 |
| --- | --- |
| `visible` | 普通背景、任务要求、标准话术等通用知识 |
| `hidden` | 客户心理、真实顾虑、隐藏异议，主要用于 AI 客户扮演 |
| `scoring_only` | 评分标准、命中点、扣分点，主要用于训练结束评分 |

注意：这是模型使用提示，不是一套用户权限系统。一期不做“谁能看、谁不能看”的前端权限控制。

### 3.6 销售训练资料入库后保存了什么

MySQL `training_knowledge_batches` 保存上传批次：

| 字段 | 中文含义 |
| --- | --- |
| `batch_id` | 批次编号 |
| `document_id` | 关联 `documents.document_id` |
| `source_type` | 来源类型 |
| `source_file` | 历史兼容文件名，新数据优先读取 `documents.filename` |
| `file_path` | 历史兼容文件路径，新数据优先读取 `documents.file_path` |
| `file_md5` | 历史兼容文件 MD5，新数据优先读取 `documents.file_md5` |
| `version_group_id` | 版本组 ID，同一个 `source_type + source_file` 多次发布会归到同一组 |
| `version_no` | 版本号，从 1 递增 |
| `previous_batch_id` | 上一个版本批次 ID |
| `is_current` | 是否为当前参与训练检索的版本 |
| `profile_type` | 历史兼容字段，新上传资料不再写入 |
| `task_type` | 历史兼容字段，新上传资料不再写入 |
| `industry` | 历史兼容字段，新上传资料不再写入 |
| `difficulty` | 历史兼容字段，新上传资料不再写入 |
| `visibility_default` | 历史兼容字段，新上传资料由后端固定写入 `visible` |
| `status` | 批次状态 |
| `chunk_count` | 切片数量 |
| `point_count` | 向量点数量 |
| `error_message` | 失败原因 |
| `quality_report_json` | 切片质量报告 |
| `created_by` | 上传人 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

销售训练资料批次状态字典：

| 状态值 | 中文名 | 说明 |
| --- | --- | --- |
| `parsing` | 解析中 | 文件已保存，正在解析和切片 |
| `pending_review` | 待确认 | 切片和质量评估已完成，等待人工确认发布 |
| `embedding` | 发布中 | 正在从临时向量库发布到正式向量库 |
| `published` | 已发布 | 资料已完成向量入库，可以参与训练检索 |
| `archived` | 历史版本 | 同版本组已有新版本发布，该版本不参与训练检索，可回滚 |
| `parsing_failed` | 解析失败 | 文件解析或临时向量库写入失败 |
| `publish_failed` | 发布失败 | 从临时向量库发布到正式向量库失败，可重试发布 |
| `deleted` | 已删除 | 批次已软删除，相关 Qdrant 向量点会被删除 |
| `duplicated` | 重复复用 | 上传响应状态，表示 MD5 命中已发布批次并复用历史数据 |

MySQL 不再保存销售训练切片正文。旧的 关系型切片正文表已取消，后端启动初始化时会清理该表；旧资料需要重新上传并重新发布。

Qdrant `sales_training_cases_staging` 保存待审核向量点；`sales_training_cases` 保存已发布和历史版本向量点。metadata：

| metadata 字段 | 中文含义 |
| --- | --- |
| `batch_id` | 上传批次编号 |
| `document_id` | 来源文件编号 |
| `chunk_id` | 切片编号 |
| `content_type` | 固定 `sales_training_case` |
| `source_type` | 来源类型 |
| `source_file` | 来源文件 |
| `file_md5` | 文件 MD5 |
| `status` | `pending_review` / `published` / `archived` |
| `is_current` | 是否当前生效版本 |
| `version_group_id` | 版本组 |
| `version_no` | 版本号 |
| `case_part` | 业务片段类型 |
| `visibility` | 可见范围 |
| `case_title` | 案例标题，LMS 策略会写入 |
| `case_index` | 案例序号，LMS 策略会写入 |

## 4. 上传后的校验方式

### 4.1 普通知识库校验

上传后应检查：

| 检查项 | 正常结果 |
| --- | --- |
| 文件列表 | `/knowledge/files` 能看到该文件 |
| 文件状态 | `status=indexed` |
| 分片数量 | `chunk_count > 0` |
| Qdrant collection | `/health` 的 `collection_points` 对应 collection 点数增加 |
| 文件预览 | `/knowledge/files/{document_id}/preview` 可查看文本，`docx` 当前除外 |
| 聊天验证 | 用文件里的问题提问，能召回并回答 |

### 4.2 销售训练资料校验

上传后应检查：

| 检查项 | 正常结果 |
| --- | --- |
| 批次列表 | `/training/knowledge/batches` 能看到该批次 |
| 批次状态 | `status=published` |
| 切片数量 | `chunk_count > 0` |
| 向量点数量 | `point_count > 0` |
| 切片列表 | `/training/knowledge/batches/{batch_id}/chunks` 能看到切片 |
| visibility 分布 | LMS 案例通常应有 `visible`、`hidden`、`scoring_only` |
| AI 客户生成 | 生成角色时能引用训练资料中的案例事实 |

## 5. 删除和重建

### 5.1 普通知识库删除

接口：

```http
DELETE /knowledge/files/{document_id}
```

删除行为：

| 层级 | 动作 |
| --- | --- |
| Qdrant | 按 `metadata.document_id` 删除向量点 |
| MySQL | `documents.status` 标记为 `deleted` |
| 原始文件 | 当前保留在 `uploads/{document_id}/`，便于排查和审计 |

### 5.2 普通知识库重建

单文件重建：

```http
POST /knowledge/files/{document_id}/reindex
```

全量重建：

```http
POST /knowledge/files/reindex-all
```

全量重建会清空并重建当前涉及的 Qdrant collection，然后重新读取 MySQL `documents` 中的 active 文件重新写入向量。文件多时会较慢，也会消耗 embedding 调用。

### 5.3 销售训练资料删除

接口：

```http
DELETE /training/knowledge/batches/{batch_id}
```

删除行为：

| 层级 | 动作 |
| --- | --- |
| Qdrant | 按 `metadata.batch_id` 删除向量点 |
| MySQL | `training_knowledge_batches.status` 标记为 `deleted` |
| 文件台账 | 关联的 `documents.status` 同步标记为 `deleted` |
| 原始文件 | 当前保留，便于审计 |

销售训练资料支持未发布批次重新切分：

```http
POST /training/knowledge/batches/{batch_id}/reparse?use_llm_fallback=true
```

已发布批次不能直接重切，避免绕过人工确认并破坏临时库到正式库的发布边界。需要修改已发布资料时，建议重新上传修正后的同名文件形成新版本，或通过版本链回滚到历史版本。

## 6. 常见问题

### 6.1 上传后状态是 duplicate / duplicated

原因：文件内容 MD5 已存在。

处理：

| 场景 | 处理方式 |
| --- | --- |
| 普通知识库，同一 collection 已存在 | 不重复入库，直接复用已有 document |
| 普通知识库，希望进入其他 collection | 上传确认时选择新 collection |
| 销售训练资料，已发布批次存在 | 直接复用历史 batch，不重复写向量 |

### 6.2 上传后 chunk_count 是 0

常见原因：

| 原因 | 解决 |
| --- | --- |
| 文件没有可抽取文本 | 检查 PDF 是否扫描件，必要时 OCR |
| DOCX 内容在图片或文本框里 | 转成普通段落文本 |
| 切分策略选错 | 普通知识库重新上传或 reindex 时选择合适策略 |
| 文件内容为空 | 补充正文 |

### 6.3 100问、FAQ 查询不准

优先检查：

| 检查项 | 推荐值 |
| --- | --- |
| `document_type` | `qa` |
| `split_strategy` | `numbered_qa` 或 `outline_qa` |
| Qdrant metadata | 应有 `question_no`、`question`、`qa_id` |
| 用户问题 | 不要求完全一致，但问题文本应能语义匹配 |

如果 PDF 没有规范书签目录，不建议使用 `outline_qa`，应转成 TXT/DOCX 的编号问答后用 `numbered_qa`。

### 6.4 销售训练 AI 客户没有引用上传资料

优先检查：

| 检查项 | 正常情况 |
| --- | --- |
| 资料是否传到训练库 | 在销售训练页历史批次中可见 |
| collection 是否正确 | 销售训练固定 `sales_training_cases` |
| 批次状态 | `published` |
| visibility | 角色生成会用 `visible` 和 `hidden` |
| LMS 结构 | 应能拆出案例、任务、话术、隐藏心理、评分标准 |

### 6.5 为什么普通上传要先预览再确认

因为普通知识库需要用户确认文档结构和切分策略。如果策略选错，FAQ、编号条目、普通文本会被切得不合理，后续向量召回会变差。

销售训练资料现在也采用“上传预览 -> 人工确认发布”的两段式流程。上传阶段只保存原文件、切片和质量报告；确认发布阶段才依赖 Qdrant 和 embedding 服务。

## 7. 推荐上传规范

### 7.1 普通知识库 FAQ 推荐格式

```text
# 一、基础问题

1. 第一个问题是什么？
答案内容。

2. 第二个问题是什么？
答案内容。

# 二、使用问题

3. 第三个问题是什么？
答案内容。
```

建议：

| 项目 | 建议 |
| --- | --- |
| 每个问题 | 独立编号 |
| 问题行 | 尽量包含问号 |
| 答案 | 紧跟问题后面 |
| 分类 | 用 Markdown 标题 |
| 文件类型 | 优先 TXT 或 DOCX |

### 7.2 普通知识库条目指南推荐格式

```text
# 选购指南

1. 小户型优先选择基础激光导航机型。
2. 有宠物家庭关注防缠绕和大吸力。
3. 地毯多的家庭关注地毯识别和自动增压。
```

建议使用 `document_type=numbered`、`split_strategy=numbered_segments`。

### 7.3 销售训练 LMS 推荐格式

```text
一、印度代理商初次沟通案例

客户案例：
客户是印度工程机械经销商，当前处于初步接触阶段。

任务要求：
学员需要完成需求挖掘，确认客户对产品和服务合作的真实关注点。

匹配答案：
可以先确认客户当前渠道、采购频次、售后能力，再介绍合作模式。

隐性心理：
客户担心交付周期不稳定，也担心售后问题影响本地口碑。

评分标准：
命中客户行业背景加分；直接报价且未挖需求扣分。
```

这种格式最适合当前 `lms_case` 策略。

## 8. 当前限制和后续可优化点

| 限制 | 当前状态 | 建议优化 |
| --- | --- | --- |
| 普通知识库 docx 预览 | 能入库，但正式预览接口暂未支持 docx | 给 `/knowledge/files/{document_id}/preview` 增加 docx 文本预览 |
| 销售训练非 LMS 资料 | 走通用策略，结构化较弱 | 为 `product_doc`、`faq`、`competitor`、`success_case` 分别增加 Strategy |

| 上传文件大小 | 当前未在接口层显式限制 | 增加大小限制和前端提示 |
| 扫描 PDF | 无 OCR | 增加 OCR 或明确要求用户上传可复制文本 |

## 9. 使用到的设计模式

### 9.1 工厂方法模式

位置：

- `rag/file_processors/factory.py`
- `training/factories/knowledge_ingest_strategy_factory.py`

原因：

不同文件类型、不同训练资料类型需要不同处理器。上传主流程不应该关心具体实现，只根据文件后缀或 `source_type` 取得对应处理器。

### 9.2 策略模式

位置：

- `rag/file_processors/*.py`
- `rag/split_strategies/*.py`
- `training/strategies/knowledge_ingest_strategy.py`

原因：

TXT、PDF、DOCX 读取方式不同；FAQ、编号条目、普通文本切分方式不同；LMS 案例和通用训练资料切片方式不同。把这些算法封装成策略，主流程可以稳定复用。

### 9.3 外观模式

位置：

- `/knowledge/upload/preview`
- `/knowledge/upload/confirm`
- `/training/knowledge/upload`

原因：

一个上传接口背后封装了文件保存、MD5 去重、结构识别、切片、embedding、写 Qdrant、写 MySQL、状态更新等多个步骤。调用方只需要调用一个接口，不需要理解内部每个子步骤。
