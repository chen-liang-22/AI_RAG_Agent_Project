# 雪花算法主键整改设计方案

## 一、设计结论

后续新增数据的所有数据库业务主键统一使用雪花算法生成的纯数字字符串。

```text
1993619824017756160
1993619824017756161
1993619824017756162
```

本次方案明确不做以下事情：

1. 不修改现有表结构。
2. 不修改历史数据主键。
3. 不批量迁移 Qdrant payload 里的历史 ID。
4. 不迁移 MinIO 历史对象路径。
5. 不改变 API 字段名，例如 `document_id`、`batch_id`、`session_id`。

最终状态是：历史数据继续保留 `doc_xxx`、`batch_xxx`、`conv_xxx` 等格式；新增数据统一使用纯雪花数字字符串。

## 二、当前现状

当前项目主键字段主要是 `VARCHAR(64)` / SQLAlchemy `String(64)`，长度足够保存纯雪花数字字符串，不需要改字段类型。

当前业务主键大量由 `uuid.uuid4().hex` 生成，并带业务前缀：

| 业务对象 | 当前格式示例 | 主要位置 |
| --- | --- | --- |
| 文件 | `doc_{uuid}` | `api/routers/knowledge.py`、`api/services/indexing_services.py`、`training/services/sales_training_service.py` |
| 会话 | `conv_{uuid}` | `rag/knowledge_store.py` |
| 消息 | `msg_{uuid}` | `rag/knowledge_store.py` |
| 字典项 | `dict_{uuid}` | `rag/knowledge_store.py` |
| 用户 | `user_{uuid}` | `api/services/auth_services.py` |
| 考试会话 | `exam_{uuid}` | `rag/knowledge_store.py` |
| 考试题目 | `exam_q_{uuid}` | `rag/knowledge_store.py` |
| 销售训练批次 | `batch_{uuid}` | `training/repository.py`、`training/services/sales_training_service.py` |
| 训练方案 | `plan_{uuid}` | `training/repository.py` |
| 客户画像 | `profile_{uuid}` | `training/repository.py` |
| 训练目标 | `setting_{uuid}` | `training/repository.py` |
| 训练轮次 | `turn_{uuid}` | `training/repository.py` |
| 训练评分 | `score_{uuid}` | `training/repository.py` |

## 三、核心原则

### 1. 新旧数据长期兼容

历史数据不迁移，所以同一个字段中会同时存在两类值：

```text
历史数据：doc_0b80a6d80e71449dafd9f37f28f9e126
新增数据：1993619824017756160
```

查询、删除、关联、Qdrant 过滤和前端展示都必须把主键当普通字符串处理，不能依赖是否带业务前缀。

### 2. 新增主键统一入口

所有数据库业务主键必须通过统一 ID 生成器创建。

禁止新增代码继续直接使用：

```python
uuid.uuid4().hex
```

统一调用形态：

```python
from infrastructure.id_generator import new_id

document_id = new_id()
batch_id = new_id()
session_id = new_id()
```

### 3. 不引入业务前缀枚举

因为新增主键确定为纯数字字符串，所以不再设计 `IdBizType` 这类业务前缀枚举。

业务类型由字段名和表名表达：

```text
documents.document_id
training_knowledge_batches.batch_id
sales_training_sessions.session_id
```

### 4. 临时 ID 和安全凭证不纳入本次范围

本次只治理数据库业务主键。临时 ID、token、临时文件名不强制改为雪花 ID。

## 四、雪花算法设计

采用 64 位雪花 ID，最终以十进制字符串写入数据库。

位段设计：

| 位段 | 位数 | 说明 |
| --- | --- | --- |
| 符号位 | 1 | 固定为 0，保证正数 |
| 时间戳 | 41 | 当前毫秒时间减自定义 epoch |
| 机器编号 | 10 | 节点编号，范围 0-1023 |
| 毫秒序列 | 12 | 同一毫秒内递增序列，范围 0-4095 |

推荐自定义 epoch：

```text
2026-01-01T00:00:00+08:00
```

单节点理论容量：

```text
4096 个 ID / 毫秒
约 409 万个 ID / 秒
```

当前项目足够使用。

## 五、配置设计

新增配置文件：

```text
config/id_generator.yml
```

建议内容：

```yaml
# 是否启用雪花算法主键生成器。
enabled: true

# 雪花算法起始时间。固定后不要随意修改。
epoch: "2026-01-01T00:00:00+08:00"

# 本机节点编号。本地默认 0；生产多实例必须不同。
worker_id: 0

# 允许的轻微时钟回拨毫秒数。
max_clock_backward_ms: 5
```

生产环境用环境变量覆盖：

```powershell
$env:ID_GENERATOR_WORKER_ID="1"
```

部署要求：

1. 本地单机开发默认 `worker_id=0`。
2. 生产单实例可以继续使用 `worker_id=0`。
3. 生产多实例必须给每个实例配置不同 `worker_id`。
4. 不建议按 IP 自动 hash 生成 `worker_id`，因为 IP 变化和 hash 冲突会增加排查成本。

## 六、代码结构设计

新增：

```text
infrastructure/id_generator.py
```

职责：

1. 读取 `config/id_generator.yml`。
2. 读取环境变量覆盖项。
3. 校验 `worker_id` 范围。
4. 初始化进程内雪花 ID 生成器。
5. 对外提供 `new_id()`。
6. 打印中文日志，方便启动排查。

建议接口：

```python
def new_id() -> str:
    """生成数据库业务主键，返回纯雪花数字字符串。"""
```

启动日志示例：

```text

```

异常日志示例：

```text
[主键生成] 系统时间回拨超过阈值 上次毫秒=1780000000000 当前毫秒=1779999999900 回拨毫秒=100
```

## 八、不替换范围

以下 ID 不属于数据库业务主键，不纳入本次强制治理：

| 场景 | 当前形式 | 原因 |
| --- | --- | --- |
| 上传预览 ID | `tmp_{uuid}` | 临时态，不是正式业务主键 |
| refresh token | `secrets.token_urlsafe()` | 安全随机凭证，不能用可预测 ID |
| 临时文件名 | `training_{uuid}.txt` | 文件临时名，不影响数据库主键 |
| Qdrant `chunk_id` | `{batch_id}_{序号}_{类型}` | 派生 ID，依赖批次和切片序号 |
| 历史迁移兼容逻辑 | 旧 ID 读取和回填 | 用于处理历史数据，不能破坏兼容 |

## 九、并发与异常处理

### 1. 同毫秒并发

同一毫秒内使用 12 位序列号递增，序列范围是 0-4095。

### 2. 序列号溢出

如果同一毫秒内序列号超过 4095：

1. 阻塞等待下一毫秒。
2. 下一毫秒序列号归零。
3. 继续生成 ID。

### 3. 轻微时钟回拨

如果当前时间小于上次生成时间，并且回拨值小于等于 `max_clock_backward_ms`：

1. 等待回拨差值。
2. 重新读取时间。
3. 继续生成 ID。

### 4. 严重时钟回拨

如果回拨值大于 `max_clock_backward_ms`：

1. 记录中文错误日志。
2. 抛出异常。
3. 拒绝生成 ID。

这里必须硬失败。接口报错可以重试，主键重复会造成数据污染。

## 十、兼容策略

### 1. MySQL 兼容

字段类型保持不变。所有主键仍然作为字符串写入。

历史和新增数据示例：

```text
doc_0b80a6d80e71449dafd9f37f28f9e126
1993619824017756160
```

### 2. API 兼容

接口字段名保持不变，只是字段值格式变化。

```json
{
  "document_id": "1993619824017756160"
}
```

前端不能用 UUID 正则校验这些字段，只能按字符串 ID 处理。

### 3. Qdrant 兼容

payload 字段名保持不变。

```json
{
  "document_id": "1993619824017756160",
  "batch_id": "1993619824017756161"
}
```

旧向量点不迁移，查询过滤时仍按字符串精确匹配。

### 4. MinIO 兼容

新文件对象路径会自然使用纯数字 `document_id`：

```text
documents/1993619824017756160/xxx.pdf
training/1993619824017756161/xxx.docx
```

旧对象路径继续保留：

```text
documents/doc_xxx/xxx.pdf
training/doc_xxx/xxx.docx
```

删除和预览逻辑必须以 `documents.object_name` 为准，不能自己重新拼路径。

## 十三、风险和处理

| 风险 | 影响 | 处理 |
| --- | --- | --- |
| 多实例 `worker_id` 重复 | 可能生成重复主键 | 生产环境显式配置不同 `ID_GENERATOR_WORKER_ID` |
| 系统时间严重回拨 | 可能导致 ID 倒退或重复 | 超过阈值直接拒绝生成 |
| 业务代码绕过统一生成器 | 新旧规则混用 | 代码扫描和测试约束 |
| 前端把 ID 当 UUID 校验 | 新 ID 校验失败 | 前端只按字符串处理 ID |
| 历史数据和新数据格式不同 | 排查时出现两种形态 | 文档明确长期兼容，不做格式假设 |
| MinIO 路径手动拼接 | 旧路径和新路径规则混乱 | 读取 `documents.object_name`，不重新拼接 |

## 十四、最终验收标准

1. 所有新增数据库业务主键都是纯雪花数字字符串。
2. 所有历史带前缀主键仍可正常查询和删除。
3. 主键字段类型不变。
4. API 字段名不变。
5. Qdrant payload 字段名不变。
6. MinIO 历史对象路径不迁移。
7. 多线程并发生成 ID 不重复。
8. 严重时钟回拨时拒绝生成 ID。
9. 日志输出保持中文。
