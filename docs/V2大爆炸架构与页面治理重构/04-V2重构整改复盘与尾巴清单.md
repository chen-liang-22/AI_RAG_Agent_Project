# V2 重构整改复盘与尾巴清单

> 本文档用于记录本轮“大爆炸架构与页面治理重构”的实际整改范围、仍未清理的旧代码尾巴、后续建议执行顺序和验证方式。  
> 当前代码按用户要求保留在本地工作树，暂未提交。

## 1. 本轮目标

本轮目标不是简单修页面或局部补丁，而是把项目从历史迭代堆叠结构逐步迁移到 V2 分层架构：

- 后端统一走 `app_v2` 分层结构：路由层、应用服务层、仓储层、基础设施适配层。
- 前端按首页、智能客服、销售训练、问答考试等功能模块拆分，避免继续把逻辑堆在单个大文件里。
- 字典内容保留原样，但字典读写能力逐步迁移到 V2 仓储。
- MinIO 作为唯一文件持久化入口，历史本地文件处理代码逐步清理。
- 日志继续使用中文；API 字段、SSE 事件名、数据库字段名保持英文协议。
- 本次不提交代码，所有改动留在本地，方便继续调整。

## 3. 当前仍剩的大尾巴

### 尾巴 1：聊天生成链路还在旧服务里

相关文件：

- `app_v2/application/chat_service.py`
- `api/services/chat_services.py`
- `rag/services/rag_service.py`
- `rag/knowledge_store.py`

现状：

- 聊天记录列表、详情、删除已走 V2 `ConversationRepository`。
- 但一次性回答、流式回答、直连 RAG、Agent 工具链、消息保存仍复用旧 `api.services.chat_services`。
- `ChatApplicationService` 仍通过 `_get_knowledge_store()` 获取旧 `KnowledgeStore`。

风险：

- 现在不能删除 `api.services.chat_services`。
- 现在不能删除 `rag.knowledge_store`。
- RAG 链路、Agent 链路、流式链路仍混在旧函数中，不利于测试和扩展。

建议后续动作：

1. 新建 `app_v2/application/rag_service.py`。
2. 新建 `app_v2/application/chat_generation_service.py`。
3. 把 `_should_use_direct_rag`、`_stream_direct_rag`、`_stream_agent` 拆成策略类。
4. 把 `_prepare_chat_conversation`、`_save_chat_exchange` 改为调用 `ConversationRepository`。
5. 删除 V2 对 `api.services.chat_services` 的依赖后，再处理旧兼容模块。

建议使用设计模式：

- 策略模式：区分直连 RAG、Agent 工具链、后续可能的兜底模型。
- 外观模式：保留 `ChatApplicationService` 作为路由入口。
- 工厂方法模式：创建不同模型、向量检索器和回答链路。

### 尾巴 2：知识上传和索引链路仍复用旧服务

相关文件：

- `app_v2/application/knowledge_service.py`
- `api/services/upload_services.py`
- `api/services/indexing_services.py`
- `api/services/document_asset_service.py`
- `api/services/upload_preview_state.py`

现状：

- V2 `KnowledgeApplicationService` 已统一入口。
- documents 表读写已基本迁移到 `DocumentRepository`。
- 但上传预览、临时文件保存、MinIO 对象提升、模型推荐切分、索引构建、data 目录重载仍复用旧服务函数。

风险：

- 旧上传服务不能删。
- 旧索引服务不能删。
- MinIO 统一入口虽然已推进，但文件处理能力还没有完全收口到 V2 adapter/service。

建议后续动作：

1. 新建 `app_v2/application/knowledge/upload_preview_service.py`。
2. 新建 `app_v2/application/knowledge/indexing_service.py`。
3. 新建 `app_v2/application/knowledge/document_asset_service.py`。
4. 把 `_save_preview_file`、`_get_preview_file`、`_promote_preview_file`、`_delete_preview_file` 迁到 V2。
5. 把 `_index_document` 迁到 V2，并让它依赖 `DocumentRepository`、`FileStorageAdapter`、`VectorStoreAdapter`。
6. 所有入口稳定后删除旧 `api.services.upload_services`、`api.services.indexing_services` 的 V2 依赖。

建议使用设计模式：

- 适配器模式：MinIO、本地临时对象、Qdrant 向量库。
- 策略模式：不同切分策略，如 QA、编号段落、递归切分。
- 模板方法模式：统一“读文件 -> 切片 -> 入向量库 -> 更新 documents 状态”的索引流程。

### 尾巴 3：销售训练核心仍是旧大类

相关文件：

- `training/services/sales_training_service.py`
- `app_v2/application/training/*.py`
- `app_v2/infrastructure/repositories/training_repository.py`

现状：

- V2 已有训练资料、画像、目标、方案、会话、评分服务入口。
- 但多数真实业务仍委托 `SalesTrainingService`。
- 首页也还复用 `SalesTrainingService` 的响应转换私有方法。

风险：

- `SalesTrainingService` 是当前最大的业务复杂度集中点。
- 继续叠功能会让销售训练难维护。
- 训练资料、画像生成、目标生成、会话、评分互相耦合，不利于后续加流程式训练、语音训练、人工复核。

建议后续动作：

1. `material_service.py` 接管训练资料上传、预览、发布、删除、重复校验、版本。
2. `profile_service.py` 接管客户画像、学员画像、角色生成、场景润色。
3. `goal_service.py` 接管开放式训练目标、动态轮数、目标达成/失败条件。
4. `session_service.py` 接管训练会话、每轮检索、流式/一次性回答。
5. `scoring_service.py` 接管 40 分通用评分、60 分 LLM 阶段评分、最终报告。
6. 把 `SalesTrainingService` 私有 mapper 移到 V2 独立 mapper 或 schema helper。
7. 确认 V2 覆盖后，删除旧训练服务里的重复实现。

建议使用设计模式：

- 建造者模式：生成训练角色、训练目标、评分规则。
- 策略模式：开放式训练、流程式训练；流式回答、一次性回答。
- 状态模式：训练资料状态、训练会话状态、训练阶段状态。
- 责任链模式：学员回答分析、客户追问生成、风险提醒、评分证据收集。

## 5. 建议后续执行顺序

建议不要直接删旧代码。后续按以下顺序做，风险最低：

1. **先拆销售训练大类**
   - 训练业务最复杂，也是后续功能最多的地方。
   - 拆完后页面和接口都会清爽很多。

2. **再拆知识上传和索引**
   - 统一 MinIO、Qdrant、documents 表写入。
   - 这是删除旧上传服务和索引服务的前提。

3. **再拆聊天 RAG 链路**
   - 把直连 RAG、Agent、流式输出、保存消息拆成清晰服务。
   - 这一步完成后 `api.services.chat_services` 才能删。

4. **再拆考试模块**
   - 当前只是 V2 入口挂旧路由。
   - 拆成 V2 服务后，旧 `api.routers.exam` 可以删除。

5. **最后做旧路由、旧服务、旧 KnowledgeStore 清理**
   - 删除前先跑引用扫描。
   - 删除后跑后端全量测试和前端构建。
