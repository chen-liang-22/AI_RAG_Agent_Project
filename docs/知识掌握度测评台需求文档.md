# 知识掌握度测评台需求文档

## 1. 背景

当前前端“问答考试”页面的生成逻辑仍然依赖知识库文件预览接口：

1. 前端调用 `/knowledge/files/{document_id}/preview` 读取原始文件文本。
2. 前端用正则从文本中拆出问题和参考答案。
3. 前端随机抽取若干题展示给用户。
4. 用户提交后，前端逐题调用 `/chat` 让大模型评分。

这个方案可以快速验证功能，但它没有真正复用 RAG 项目已经具备的结构化知识能力。知识库入库阶段已经完成了 `qa`、`numbered`、`outline_qa` 等切分，并且在 Qdrant payload 中保存了 `question_id`、`section_path`、`source_page`、`document_type`、`split_strategy` 等 metadata。考试模块继续读取原始文件并正则拆分，会导致题目质量、可控性、评分稳定性都比较弱。

因此，考试模块建议从“文件预览抽题”升级为“知识掌握度测评台”。

## 2. 产品定位

知识掌握度测评台不是简单生成一张试卷，而是围绕知识库评估用户对知识点的掌握情况。

它应该支持：

- 按知识库、文件、章节、题型、难度生成测评。
- 从结构化知识单元生成题目，而不是从原始文件正则猜题。
- 基于参考知识进行结构化评分。
- 记录错题、薄弱点和历史测评结果。
- 支持后续扩展为面试训练、错题复训、能力雷达图和学习路径推荐。

## 3. 目标

### 3.1 业务目标

- 提升考试题目的准确性和可控性。
- 让知识库切分、向量检索、metadata 能在考试场景中真正发挥作用。
- 支持用户按章节和知识点进行定向训练。
- 通过评分结果沉淀用户薄弱点，为后续个性化复习提供依据。

### 3.2 技术目标

- 取消前端 `extractQuestions()` 正则抽题逻辑。
- 前端不再使用预览接口生成试卷。
- 后端新增考试专用接口，统一负责组卷、题目生成和评分。
- 评分接口返回结构化 JSON，而不是只有自然语言点评。
- 评分支持批量处理，避免前端逐题串行调用 `/chat`。

## 4. 当前问题

### 4.1 题源不稳定

当前题目来自原始文本预览，依赖正则识别：

- 文本格式稍有变化就可能抽不出题。
- PDF 转文本后的换行、编号、页眉页脚会干扰解析。
- `outline_qa` 这类目录型问答文档的结构信息没有被使用。

### 4.2 没有利用向量库结构

当前考试逻辑没有使用：

- Qdrant payload。
- `document_id` 过滤。
- `question_id`。
- `section_path`。
- `source_page`。
- `document_type`。
- `split_strategy`。

这会让考试模块和知识库索引状态脱节。

### 4.3 组卷方式单一

当前只有“随机抽若干题”，缺少：

- 章节测验。
- 难度分布。
- 题型分布。
- 知识点覆盖率控制。
- 错题复训。
- 面试追问模式。

### 4.4 评分结果不可沉淀

当前评分结果是一段自然语言，后续很难统计：

- 哪些要点命中。
- 哪些要点缺失。
- 哪些知识点薄弱。
- 单题得分。
- 总得分。
- 是否需要追问。

## 5. 新方案概述

新方案将考试模块拆成三个核心能力：

1. 题源选择：从知识库结构化单元中选择可出题内容。
2. 智能组卷：按策略生成或抽取题目。
3. 结构化评分：基于参考知识给出得分、扣分点和复习建议。

推荐流程：

```text
选择知识库/文件/章节
        ↓
读取结构化知识单元
        ↓
按组卷策略生成题目
        ↓
用户答题
        ↓
批量结构化评分
        ↓
生成测评报告和错题记录
```

## 6. 核心模式

### 6.1 随机练习模式

适合快速自测。

用户选择：

- 知识库 collection。
- 文件或全部文件。
- 题目数量。
- 题型。

系统行为：

- 从满足条件的知识单元中随机抽题。
- 优先使用已有 `qa` 知识单元。
- 如果题源不足，再基于相关知识 chunk 调用 LLM 生成题目。

### 6.2 章节测验模式

适合 Java 面试题、课程讲义、产品手册这类有目录层级的资料。

用户选择：

- 文件。
- 章节或目录路径。
- 题目数量。
- 难度。

系统行为：

- 按 `section_path` 或 `category` 过滤知识单元。
- 保证题目来自当前章节。
- 评分时展示来源章节和页码。

### 6.3 面试训练模式

适合 Java 面试题、客服培训、产品知识考核。

系统行为：

- 先给出一个主问题。
- 用户回答后，大模型根据回答质量和参考知识生成追问。
- 追问最多 N 轮。
- 最终给出综合评分。

特点：

- 比普通试卷更接近真实面试。
- 可以检测用户是否只是背答案，还是理解原理。

### 6.4 薄弱点复训模式

适合已经有历史答题记录的用户。

系统行为：

- 统计用户历史错题。
- 按知识点、章节、题型聚合薄弱项。
- 优先生成相关题目。
- 用户答对后降低该知识点权重。

### 6.5 混合测评模式

适合正式测验。

支持配置：

- 简答题数量。
- 单选题数量。
- 多选题数量。
- 判断题数量。
- 场景分析题数量。
- 难度比例。

示例：

```text
共 10 题：
- 3 道概念题
- 2 道对比题
- 2 道场景题
- 2 道排错题
- 1 道开放追问题
```

## 7. 题型设计

### 7.1 简答题

来源：

- 直接使用 `qa` 的 question。
- 或基于 chunk 生成新的问题。

评分重点：

- 核心概念是否准确。
- 关键步骤是否完整。
- 是否有明显错误。

### 7.2 单选题

来源：

- 由 LLM 基于知识单元生成。

要求：

- 只有一个正确答案。
- 干扰项必须合理，但不能违背参考知识。
- 必须保存正确选项和解析。

### 7.3 多选题

来源：

- 适合特性列表、注意事项、配置项、故障原因等知识点。

要求：

- 多个正确答案。
- 评分支持部分得分。

### 7.4 判断题

来源：

- 适合概念辨析和常见误区。

要求：

- 必须给出判断依据。

### 7.5 场景分析题

来源：

- 由多个相关 chunk 组合生成。

要求：

- 题目描述一个真实使用场景。
- 用户需要给出分析过程和处理建议。
- 评分重点是思路，而不只是结论。

### 7.6 面试追问题

来源：

- 基于用户上一轮回答动态生成。

要求：

- 追问必须围绕当前题目和参考知识。
- 不允许跳到无关知识点。

## 8. 后端接口设计

### 8.1 生成试卷接口

```http
POST /exam/generate
```

请求示例：

```json
{
  "collection_name": "agent",
  "document_id": "doc_xxx",
  "section_path": "Java集合/泛型面试题",
  "mode": "chapter_test",
  "question_count": 10,
  "question_types": ["short_answer", "single_choice"],
  "difficulty": "medium",
  "source_policy": "structured_first"
}
```

字段说明：

- `collection_name`：知识库 collection。
- `document_id`：指定题源文件，可为空。
- `section_path`：指定章节，可为空。
- `mode`：组卷模式。
- `question_count`：题目数量。
- `question_types`：题型列表。
- `difficulty`：难度。
- `source_policy`：题源策略。

响应示例：

```json
{
  "paper_id": "paper_xxx",
  "title": "Java集合章节测验",
  "total_score": 100,
  "questions": [
    {
      "question_id": "exam_q_001",
      "source_question_id": "doc_xxx_outline_q_0001",
      "question_type": "short_answer",
      "difficulty": "medium",
      "question": "HashMap 和 Hashtable 的区别是什么？",
      "options": [],
      "reference_answer": "HashMap 线程不安全，Hashtable 线程安全...",
      "score": 10,
      "source": {
        "document_id": "doc_xxx",
        "filename": "Java面试题.pdf",
        "section_path": "Java集合/泛型面试题 / HashMap和HashTable的区别",
        "source_page": 19
      }
    }
  ]
}
```

### 8.2 批量评分接口

```http
POST /exam/grade
```

请求示例：

```json
{
  "paper_id": "paper_xxx",
  "user_id": "1002",
  "answers": [
    {
      "question_id": "exam_q_001",
      "answer": "HashMap 不安全，Hashtable 安全..."
    }
  ],
  "model_mode": "high"
}
```

响应示例：

```json
{
  "paper_id": "paper_xxx",
  "total_score": 86,
  "max_score": 100,
  "results": [
    {
      "question_id": "exam_q_001",
      "score": 8,
      "max_score": 10,
      "hit_points": ["说出了线程安全差异", "提到了性能差异"],
      "missing_points": ["没有说明 null key/null value 支持差异"],
      "wrong_points": [],
      "comment": "回答覆盖了核心差异，但细节不完整。",
      "review_suggestion": "复习 HashMap、Hashtable、ConcurrentHashMap 的对比。"
    }
  ],
  "weak_points": [
    {
      "knowledge_point": "Map 实现类对比",
      "section_path": "Java集合/泛型面试题",
      "error_count": 2
    }
  ]
}
```

### 8.3 章节列表接口

```http
GET /exam/sources
```

用途：

- 返回可用于考试的文件、章节、题源数量。
- 前端用于展示章节树和可出题数量。

响应示例：

```json
{
  "collections": [
    {
      "collection_name": "agent",
      "documents": [
        {
          "document_id": "doc_xxx",
          "filename": "Java面试题.pdf",
          "question_count": 1234,
          "sections": [
            {
              "section_path": "Java集合/泛型面试题",
              "question_count": 80
            }
          ]
        }
      ]
    }
  ]
}
```

### 8.4 错题记录接口

```http
GET /exam/mistakes
POST /exam/mistakes/retry
```

用途：

- 查询用户错题。
- 基于错题生成复训题。
- 支持后续薄弱点模式。

## 9. 题源策略

### 9.1 structured_first

优先使用结构化问答单元。

适用：

- `document_type = qa`
- `split_strategy = qa`
- `split_strategy = outline_qa`

优点：

- 快。
- 稳定。
- 不需要额外 LLM 出题成本。

### 9.2 llm_generate

基于召回的知识 chunk 生成新题。

适用：

- 普通 `text` 文档。
- `numbered` 文档。
- 需要生成选择题或场景题时。

优点：

- 题型更丰富。
- 可以控制难度和题目风格。

风险：

- 需要严格绑定参考知识。
- 需要校验题目和答案是否都能从参考知识中找到依据。

### 9.3 mixed

结构化题源和 LLM 生成混合使用。

适用：

- 正式测评。
- 题源数量不足。
- 想避免每次题目完全重复。

## 11. 前端页面设计

### 11.1 页面定位

页面名称建议改为：

```text
知识掌握度测评台
```

### 11.2 页面布局

建议采用三栏布局：

```text
左侧：组卷配置
中间：答题区
右侧：测评结果 / 知识点分析
```

### 11.3 左侧组卷配置

包含：

- Collection 选择。
- 题源文件选择。
- 章节树选择。
- 组卷模式。
- 题型选择。
- 难度选择。
- 题目数量。
- 单题分值。
- 生成按钮。

### 11.4 中间答题区

包含：

- 题目列表。
- 单选/多选组件。
- 简答输入框。
- 当前题来源提示。
- 查看参考答案开关。
- 提交评分按钮。

### 11.5 右侧分析区

包含：

- 总得分。
- 命中要点。
- 缺失要点。
- 错误点。
- 薄弱章节。
- 复习建议。
- 重新练习按钮。

## 12. 日志要求

日志输出必须使用中文。

建议关键日志：

```text
[考试] 开始生成试卷 用户编号=1002 模式=chapter_test 题目数量=10
[考试] 题源筛选完成 collection=agent 文件编号=doc_xxx 章节=Java集合/泛型面试题 候选题数=80
```

## 13. 权限和边界

- 只能从当前系统已入库知识中生成题目。
- LLM 生成题目时必须绑定参考 chunk。
- 评分时不得引入参考知识以外的专业结论。
- 如果参考知识不足，必须标记为“参考知识不足”，不能强行评分。
- 用户答案和评分记录需要和 `user_id` 关联。

## 14. 迭代计划

### 第一阶段：替换当前抽题逻辑

目标：

- 去掉前端正则抽题。
- 新增 `/exam/generate`。
- 从结构化 QA 中生成简答题。
- 新增 `/exam/grade` 批量评分。

范围：

- 只支持简答题。
- 只支持随机练习和章节测验。
- 保存试卷和评分结果可以先不落库，先返回给前端展示。

### 第二阶段：题型扩展

目标：

- 支持单选、多选、判断、场景分析。
- 支持题型比例。
- 支持难度参数。

范围：

- LLM 生成题目。
- 增加题目质量校验。
- 增加来源 chunk 展示。

### 第三阶段：记录和复训

目标：

- 保存答题记录。
- 保存错题记录。
- 生成薄弱点统计。
- 支持错题复训。

范围：

- 新增考试相关 SQLite 表。
- 新增错题接口。
- 前端增加测评报告和错题本。

### 第四阶段：面试训练

目标：

- 支持多轮追问。
- 支持面试官风格评分。
- 输出综合评价。

范围：

- 新增面试会话状态。
- 每轮追问绑定参考知识。
- 输出能力维度评分。

## 15. 推荐优先级

优先实现：

1. `/exam/generate`
2. `/exam/grade`
3. 前端移除 `extractQuestions()`
4. 批量评分
5. 章节测验

暂缓实现：

- 能力雷达图。
- 错题本。
- 面试追问。
- 多题型混合卷。

原因：

第一阶段先解决当前考试模块的根问题：题源不稳定、评分慢、结构化知识没有被使用。后续再扩展体验和数据沉淀。
