# 知识掌握度测评台需求文档

## 1. 背景

当前前端“问答考试”页面的生成逻辑仍然依赖知识库文件预览接口：

1. 前端调用 `/knowledge/files/{document_id}/preview` 读取原始文件文本。
2. 前端用正则从文本中拆出问题和参考答案。
3. 前端随机抽取若干题展示给用户。
4. 用户提交后，前端逐题调用 `/chat` 让大模型评分。

这个方案可以快速验证功能，但它没有真正复用 RAG 项目已经具备的结构化知识能力。知识库入库阶段已经完成了 `qa`、`numbered`、`outline_qa` 等切分，并且在 Qdrant payload 中保存了 `question_id`、`section_path`、`source_page`、`document_type`、`split_strategy` 等 metadata。考试模块继续读取原始文件并正则拆分，会导致题目质量、可控性、评分稳定性都比较弱。

因此，考试模块建议从“文件预览抽题”升级为“知识掌握度测评台”。

## 2. 产品定位

知识掌握度测评台不是简单生成一张试卷，而是围绕知识库评估用户对知识点的掌握情况。

它应该支持：

- 按知识库、文件、章节、题型、难度生成测评。
- 从结构化知识单元生成题目，而不是从原始文件正则猜题。
- 基于参考知识进行结构化评分。
- 记录错题、薄弱点和历史测评结果。
- 支持后续扩展为面试训练、错题复训、能力雷达图和学习路径推荐。

## 3. 目标

### 3.1 业务目标

- 提升考试题目的准确性和可控性。
- 让知识库切分、向量检索、metadata 能在考试场景中真正发挥作用。
- 支持用户按章节和知识点进行定向训练。
- 通过评分结果沉淀用户薄弱点，为后续个性化复习提供依据。

### 3.2 技术目标

- 取消前端 `extractQuestions()` 正则抽题逻辑。
- 前端不再使用预览接口生成试卷。
- 后端新增考试专用接口，统一负责组卷、题目生成和评分。
- 评分接口返回结构化 JSON，而不是只有自然语言点评。
- 评分支持批量处理，避免前端逐题串行调用 `/chat`。

## 4. 当前问题

### 4.1 题源不稳定

当前题目来自原始文本预览，依赖正则识别：

- 文本格式稍有变化就可能抽不出题。
- PDF 转文本后的换行、编号、页眉页脚会干扰解析。
- `outline_qa` 这类目录型问答文档的结构信息没有被使用。

### 4.2 没有利用向量库结构

当前考试逻辑没有使用：

- Qdrant payload。
- `document_id` 过滤。
- `question_id`。
- `section_path`。
- `source_page`。
- `document_type`。
- `split_strategy`。

这会让考试模块和知识库索引状态脱节。

### 4.3 组卷方式单一

当前只有“随机抽若干题”，缺少：

- 章节测验。
- 难度分布。
- 题型分布。
- 知识点覆盖率控制。
- 错题复训。
- 面试追问模式。

### 4.4 评分结果不可沉淀

当前评分结果是一段自然语言，后续很难统计：

- 哪些要点命中。
- 哪些要点缺失。
- 哪些知识点薄弱。
- 单题得分。
- 总得分。
- 是否需要追问。

## 5. 新方案概述

新方案将考试模块拆成三个核心能力：

1. 题源选择：从知识库结构化单元中选择可出题内容。
2. 智能组卷：按策略生成或抽取题目。
3. 结构化评分：基于参考知识给出得分、扣分点和复习建议。

推荐流程：

```text
选择知识库/文件/章节
        ↓
读取结构化知识单元
        ↓
按组卷策略生成题目
        ↓
用户答题
        ↓
批量结构化评分
        ↓
生成测评报告和错题记录
```

## 6. 核心模式

### 6.1 随机练习模式

适合快速自测。

用户选择：

- 知识库 collection。
- 文件或全部文件。
- 题目数量。
- 题型。

系统行为：

- 从满足条件的知识单元中随机抽题。
- 优先使用已有 `qa` 知识单元。
- 如果题源不足，再基于相关知识 chunk 调用 LLM 生成题目。

### 6.2 章节测验模式

适合 Java 面试题、课程讲义、产品手册这类有目录层级的资料。

用户选择：

- 文件。
- 章节或目录路径。
- 题目数量。
- 难度。

系统行为：

- 按 `section_path` 或 `category` 过滤知识单元。
- 保证题目来自当前章节。
- 评分时展示来源章节和页码。

### 6.3 面试训练模式

适合 Java 面试题、客服培训、产品知识考核。

系统行为：

- 先给出一个主问题。
- 用户回答后，大模型根据回答质量和参考知识生成追问。
- 追问最多 N 轮。
- 最终给出综合评分。

特点：

- 比普通试卷更接近真实面试。
- 可以检测用户是否只是背答案，还是理解原理。

### 6.4 薄弱点复训模式

适合已经有历史答题记录的用户。

系统行为：

- 统计用户历史错题。
- 按知识点、章节、题型聚合薄弱项。
- 优先生成相关题目。
- 用户答对后降低该知识点权重。

### 6.5 混合测评模式

适合正式测验。

支持配置：

- 简答题数量。
- 单选题数量。
- 多选题数量。
- 判断题数量。
- 场景分析题数量。
- 难度比例。

示例：

```text
共 10 题：
- 3 道概念题
- 2 道对比题
- 2 道场景题
- 2 道排错题
- 1 道开放追问题
```

## 7. 题型设计

### 7.1 简答题

来源：

- 直接使用 `qa` 的 question。
- 或基于 chunk 生成新的问题。

评分重点：

- 核心概念是否准确。
- 关键步骤是否完整。
- 是否有明显错误。

### 7.2 单选题

来源：

- 由 LLM 基于知识单元生成。

要求：

- 只有一个正确答案。
- 干扰项必须合理，但不能违背参考知识。
- 必须保存正确选项和解析。

### 7.3 多选题

来源：

- 适合特性列表、注意事项、配置项、故障原因等知识点。

要求：

- 多个正确答案。
- 评分支持部分得分。

### 7.4 判断题

来源：

- 适合概念辨析和常见误区。

要求：

- 必须给出判断依据。

### 7.5 场景分析题

来源：

- 由多个相关 chunk 组合生成。

要求：

- 题目描述一个真实使用场景。
- 用户需要给出分析过程和处理建议。
- 评分重点是思路，而不只是结论。

### 7.6 面试追问题

来源：

- 基于用户上一轮回答动态生成。

要求：

- 追问必须围绕当前题目和参考知识。
- 不允许跳到无关知识点。

## 8. 后端接口设计

当前实现已经从“生成一张试卷再批量评分”调整为“对话式考试会话”。旧接口 `/exam/generate` 和 `/exam/grade` 已移除，前端只对接下面的会话式接口。

### 8.1 查询考试目录接口

```http
GET /exam/sections?collection_name=agent&document_id=doc_xxx
```

用途：

- 查询某个向量库和文件下可用于考试的一级目录。
- 目录只展示第一层，例如 `1、Java 基础面试题`。
- 完整目录路径仍保存在题目来源中，方便历史详情追溯。

响应示例：

```json
{
  "collection_name": "agent",
  "document_id": "doc_xxx",
  "sections": [
    {
      "section_path": "1、Java 基础面试题",
      "question_count": 42
    }
  ]
}
```

### 8.2 开始考试会话接口

```http
POST /exam/sessions
```

请求示例：

```json
{
  "collection_name": "agent",
  "document_id": "doc_xxx",
  "section_path": "1、Java 基础面试题",
  "user_id": "1002",
  "round_count": 10,
  "question_types": ["single_choice", "multiple_choice", "true_false", "short_answer", "fill_blank"],
  "model_mode": "high",
  "seed": 12345
}
```

字段说明：

- `collection_name`：题源所在 Qdrant collection。
- `document_id`：指定题源文件，可为空。
- `section_path`：指定一级目录，可为空。
- `user_id`：考试用户编号。
- `round_count`：考试轮数，当前限制为 1 到 50。
- `question_types`：随机题型范围。
- `model_mode`：命题和阅卷模型档位。
- `seed`：随机种子，可为空；填写后同一题源范围更容易复现抽题结果。

### 8.4 查询考试历史接口

```http
GET /exam/sessions?page=1&page_size=10&user_id=1002&keyword=Java
```

用途：

- 分页查询考试历史。
- 支持按 `user_id` 过滤。
- 支持按文件名、目录、会话编号等关键词搜索。

### 8.5 查询考试详情接口

```http

```

用途：

- 查看一场考试的全部题目。

- 后续错题复训和薄弱点分析可以直接基于该详情数据扩展。

## 9. 题源和命题策略

### 9.1 题源筛选

当前考试只从 Qdrant 中 `metadata.content_type = qa` 的结构化问答片段抽题。筛选顺序如下：

1. 按 `collection_name` 选择向量库。
2. 按 `document_id` 选择文件，可为空。
3. 按 `section_path` 选择一级目录，可为空。
4. 从候选题中按 `question_id`、`qa_id` 或题干去重。

### 9.2 LLM 命题

所有题型都会优先经过 LLM 润色：

- 单选题：生成 4 个选项，只有 1 个正确答案。
- 多选题：生成 4 到 6 个选项，至少 2 个正确答案。
- 判断题：固定选项为 `正确`、`错误`。
- 填空题：题干需要自然出现空缺。
- 简答题：题干和标准答案都需要像正式考试题。

LLM 生成失败时，后端会用规则兜底生成题目，保证考试流程不中断。

### 9.3 LLM 阅卷

所有题型统一由 LLM 阅卷，包括单选、多选、判断、填空和简答。阅卷需要返回：

- 得分。
- 是否正确。
- 润色后的正确答案。
- 命中点。
- 遗漏点。
- 错误点。
- 面向用户的自然点评。

LLM 阅卷失败时，后端会用基础规则兜底评分，并在点评中说明已使用规则兜底。

## 11. 前端页面设计

### 11.1 页面定位

页面名称：对话式考试 / 知识掌握度随机问答。

它不再展示传统试卷，而是按聊天形式逐轮问答。

### 11.2 页面布局

当前采用三栏布局：

```text
左侧：题源设置
中间：对话式答题区
右侧：历史记录
```

### 11.3 左侧题源设置

包含：

- 向量库选择。
- 文件选择。
- 一级目录选择。
- 轮数设置。
- 随机种子。
- 分析模型档位。
- 题型选择。
- 开始测评按钮。

### 11.4 中间对话式答题区

包含：

- 当前进度。
- 当前总分。
- 题目消息。
- 用户答案消息。
- 单题分析消息。
- 单选、多选、判断、填空、简答输入控件。

### 11.5 右侧历史记录

包含：

- 历史考试列表。
- 每页 10 条分页。
- 关键词搜索。
- 考试详情抽屉。
- 详情中展示题目、用户答案、正确答案、遗漏点和点评。

## 13. 权限和边界

- 只能从当前系统已入库的结构化 QA 片段中抽题。
- LLM 命题必须绑定原始问题和参考答案，不允许凭空引入新事实。
- 阅卷时可以接受同义表达、合理缩写和顺序差异。
- 对明显漏点、错选和事实错误要扣分。
- 每场考试必须保存会话、题目、用户答案和分析结果。
- 对外 API 字段仍使用英文，日志和注释使用中文。

## 14. 已实现能力

- 取消前端正则抽题和预览接口生成试卷。
- 移除旧 `/exam/generate` 和 `/exam/grade` 接口。
- 支持 `/exam/sections` 查询一级目录。
- 支持 `/exam/sessions` 创建对话式考试会话。
- 支持 `/exam/sessions/{session_id}/answer` 逐轮作答和评分。
- 支持 `/exam/sessions` 分页查询历史考试。
- 支持 `/exam/sessions/{session_id}` 查看考试详情。
- 支持单选、多选、判断、填空、简答五种题型。
- 所有题型都优先经过 LLM 命题和 LLM 阅卷。
- 考试记录已落 MySQL，可供后续复盘和分析。

## 15. 后续优先级

优先继续：

1. 薄弱点聚合：按目录、题型、知识点统计错误和遗漏。
