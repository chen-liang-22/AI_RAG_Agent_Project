# 合格工程架构改造设计

## 一、改造目标

本次改造目标不是重写项目，而是把当前 RAG 应用从“功能可用、可部署”提升到“边界清楚、可测试、可观测、可持续迭代”的合格工程架构。

当前项目已经具备完整 RAG 链路，包括文件上传、文本解析、切片、Embedding、Qdrant 入库、检索、LLM 回答、会话历史、销售训练和考试等业务能力。主要短板在于目录演进痕迹较重、部分 Service 职责过大、RAG 效果评估不足、入库任务同步链路偏重、监控指标还不系统。

目标评分从当前约 75 分提升到 82-85 分。

## 二、总体架构定位

改造后项目按职责划分为五层：

```text
前端 Vue3
  -> FastAPI 接口层
  -> 应用服务层 app/application
  -> 领域模型层 app/domain
  -> 基础设施层 app/infrastructure
  -> AI/RAG 基础能力 core
```

各层职责如下：

| 层级 | 目录 | 职责 |
|---|---|---|
| 启动入口 | `api/` | 只保留 FastAPI app 创建、路由挂载、CORS、健康检查、启动预热 |
| 业务接口 | `app/api/` | 定义 HTTP 路由，接收请求，调用 application 服务 |
| 应用服务 | `app/application/` | 编排业务流程，例如知识库、聊天、销售训练、考试 |
| 领域模型 | `app/domain/` | 实体、枚举、schema、业务常量 |
| 基础设施 | `app/infrastructure/` | MySQL、Redis、MinIO、Qdrant、ID 生成器、Repository |
| 通用能力 | `core/` | RAG 检索、切片、精排、模型工厂、配置、日志、Prompt 管理 |
| 配置 | `config/` | 应用配置、存储配置、训练配置、Prompt 配置 |
| 文档 | `docs/` | 架构设计、接口文档、数据库设计、部署说明 |
| 测试 | `tests/` | 单元测试、集成测试、RAG 评估测试 |

## 三、目录收敛设计

当前 `app_v2` 是历史演进目录，长期看不适合作为稳定业务目录名。建议收敛为：

```text
api/
app/
  api/
  application/
  domain/
  infrastructure/
  shared/
core/
  rag/
  model/
  utils/
config/
docs/
tests/
```

改造原则：

1. `api/` 只做启动入口，不承载复杂业务逻辑。
2. `app/` 承载业务应用层，替代当前 `app_v2/`。
3. `core/` 只保留与具体业务无关的 AI/RAG 基础能力。
4. 业务代码不能反向依赖路由层。
5. `app/application` 可以依赖 `app/domain`、`app/infrastructure`、`core`。
6. `core` 不依赖 `app/application`，避免基础能力被业务绑死。

## 四、业务 Service 拆分设计

### 4.1 销售训练模块

当前销售训练核心 Service 职责偏多，后续建议拆成：

```text
app/application/training/
  training_knowledge_service.py
  training_role_service.py
  training_session_service.py
  training_score_service.py
  training_query_service.py
```

| Service | 职责 |
|---|---|
| `training_knowledge_service.py` | 训练资料上传、解析、切片、发布、回滚、删除 |
| `training_role_service.py` | 学员画像、客户画像、补充问题、AI 客户角色生成 |
| `training_session_service.py` | 训练会话创建、对话轮次、流式回复、历史记录 |
| `training_score_service.py` | 训练评分、报告生成、评分规则兜底 |
| `training_query_service.py` | 训练资料向量检索、证据过滤、证据格式化 |

### 4.2 考试模块

考试模块建议拆成：

```text
app/application/exam/
  exam_source_service.py
  exam_session_service.py
  exam_question_service.py
  exam_grading_service.py
```

| Service | 职责 |
|---|---|
| `exam_source_service.py` | 从 Qdrant 查询结构化 QA 题源和章节目录 |
| `exam_session_service.py` | 考试会话创建、分页查询、删除、详情 |
| `exam_question_service.py` | 题目生成、题型转换、选项清洗、兜底命题 |
| `exam_grading_service.py` | 模型阅卷、规则兜底、分析结果格式化 |

### 4.3 知识库模块

知识库模块保持主线清晰：

```text
app/application/knowledge/
  knowledge_file_service.py
  knowledge_preview_service.py
  knowledge_indexing_service.py
  knowledge_delete_service.py
```

| Service | 职责 |
|---|---|
| `knowledge_file_service.py` | 文件列表、详情、预览、元数据 |
| `knowledge_preview_service.py` | 临时上传、切分策略推荐、确认入库 |
| `knowledge_indexing_service.py` | 文本解析、切片、Embedding、Qdrant 入库 |
| `knowledge_delete_service.py` | MySQL、Qdrant、MinIO、Redis 全链路删除 |

## 五、RAG 链路工程化设计

RAG 查询链路建议抽成固定 pipeline：

```text
用户问题
 -> Query Planner
 -> Retriever
 -> Reranker
 -> Context Builder
 -> Answer Generator
 -> Evaluation Logger
```

推荐目录：

```text
core/rag/pipeline/
  query_planner.py
  retriever.py
  reranker.py
  context_builder.py
  answer_generator.py
  evaluation_logger.py
```

| 组件 | 职责 |
|---|---|
| `QueryPlanner` | 根据用户问题和会话历史生成多个 search_query |
| `Retriever` | 调用 Qdrant 多路召回候选片段 |
| `Reranker` | 对候选片段精排，选择最相关上下文 |
| `ContextBuilder` | 控制最终上下文数量、长度和格式 |
| `AnswerGenerator` | 调用 LLM 生成一次性或流式回答 |
| `EvaluationLogger` | 记录召回、精排、耗时、最终回答，供评估使用 |

推荐配置：

```yaml
rag:
  per_query_top_k: 5
  per_query_keep: 2
  final_context_limit: 6
  reranker_enabled: true
  hybrid_search_enabled: false
```

## 六、RAG 评估体系设计

新增 RAG 评估集，用于判断知识库效果是否稳定。

建议文件：

```text
tests/rag_eval/
  rag_eval_cases.yml
  test_rag_retrieval_eval.py
```

评估样例结构：

```yaml
- case_id: robot_001
  question: 中国市场啥时候开始普及扫地机器人？
  expected_keywords:
    - 2015
    - 小米生态链
  expected_sources:
    - 扫地机器人100问.pdf
```

评估指标：

| 指标 | 含义 |
|---|---|
| 召回命中率 | 期望文档是否被召回 |
| 关键词命中率 | 回答是否覆盖关键事实 |
| 无依据回答率 | 是否编造知识库外内容 |
| 首字耗时 | 流式回答首个 chunk 耗时 |
| 总耗时 | 完整回答总耗时 |

## 七、异步入库设计

文件入库建议从同步流程改成任务模式：

```text
上传文件
 -> 保存 MinIO
 -> 创建 ingestion_task
 -> 后台解析文本
 -> 切片
 -> Embedding
 -> 写 Qdrant
 -> 更新任务状态
```

初期可以先用 FastAPI BackgroundTasks 或线程池，后续再升级 Celery、Dramatiq、ARQ。

建议任务状态：

| 状态 | 含义 |
|---|---|
| `pending` | 已创建，等待处理 |
| `parsing` | 正在解析文件 |
| `chunking` | 正在切片 |
| `embedding` | 正在向量化 |
| `indexing` | 正在写入 Qdrant |
| `completed` | 入库成功 |
| `failed` | 入库失败 |
| `deleted` | 已删除 |

## 八、全链路删除设计

知识库和销售训练资料删除必须统一处理：

```text
删除入口
 -> 查 MySQL 文档和关联资料
 -> 删除 Qdrant points
 -> 删除 MinIO 文件
 -> 删除 Redis 临时状态
 -> 删除 MySQL 业务记录
 -> 记录删除结果
```

需要注意：

1. MySQL 删除建议放在最后，避免文件和向量删除失败后丢失索引依据。
2. Qdrant 和 MinIO 删除失败时要记录日志，并允许后续补偿。
3. 删除接口返回需要说明每个资源的删除结果。
4. 训练资料和知识库资料如果共用 documents 表，删除时要清理两侧关联数据。

## 九、可观测性设计

日志继续保持中文，但要结构化记录关键字段。

建议统一字段：

| 字段 | 含义 |
|---|---|
| `trace_id` | 一次请求链路编号 |
| `conversation_id` | 会话编号 |
| `document_id` | 文档编号 |
| `collection_name` | Qdrant collection |
| `model_name` | 模型名称 |
| `query_count` | Query Planner 拆分数量 |
| `retrieved_count` | 召回数量 |
| `reranked_count` | 精排保留数量 |
| `first_token_ms` | 首字耗时 |
| `total_ms` | 总耗时 |
| `error` | 错误信息 |

后续可增加 Prometheus 指标：

```text
rag_request_total
rag_request_failed_total
rag_retrieval_duration_seconds
rag_llm_first_token_seconds
rag_llm_total_seconds
document_ingestion_total
document_ingestion_failed_total
```

## 十、配置治理设计

配置继续拆成：

```text
config/app.yml
config/storage.yml
config/training.yml
config/prompts.yml
```

职责如下：

| 文件 | 职责 |
|---|---|
| `app.yml` | 应用运行参数、RAG 参数、Agent 参数 |
| `storage.yml` | MySQL、Redis、MinIO、Qdrant |
| `training.yml` | 销售训练业务配置 |
| `prompts.yml` | 所有发给大模型的 prompt |

建议后续增加 Pydantic Settings 或配置校验器，启动时校验必填配置，避免运行到接口调用时才报错。

## 十一、设计模式使用

| 设计模式 | 使用位置 | 选择原因 |
|---|---|---|
| 外观模式 | `KnowledgeService`、`TrainingService`、`ExamService` | 对接口层隐藏 MySQL、Qdrant、MinIO、LLM 的复杂组合 |
| 策略模式 | 切片策略、检索策略、资料解析策略 | 不同文档类型和业务场景可以替换算法 |
| 工厂模式 | 模型工厂、文件解析器工厂、切片策略工厂 | 根据配置或文件类型创建不同实现 |
| 模板方法模式 | 文件入库流程 | 入库主流程固定，具体解析和切片可扩展 |
| 责任链模式 | RAG 查询 pipeline | Query Planner、召回、精排、上下文构造、回答生成按顺序处理 |

## 十二、边界与不做事项

本次设计不建议马上做：

1. 不马上引入知识图谱，避免一次改造过大。
2. 不马上替换 Qdrant，当前向量库仍然合适。
3. 不马上改前端大页面，只配合后端新状态字段做必要展示。
4. 不一次性重写所有业务，优先按阶段拆分。
5. 不删除旧功能入口，确认新路径稳定后再清理。

## 十三、预期收益

完成改造后：

1. 新人能快速看懂目录职责。
2. 单个 Service 文件变小，维护成本下降。
3. RAG 链路可独立测试和评估。
4. 大文件入库不会长时间卡接口。
5. 删除链路更可靠，减少 MySQL、Qdrant、MinIO 数据不一致。
6. 日志和指标能定位慢点、失败点和召回质量问题。

整体架构可从“可部署 RAG 应用”提升为“合格工程化 RAG 架构”。
