# 异步入库与销售资料交互设计

## 一、目标

本设计用于替代 XXL-JOB 方案，改为项目内部异步入库。

核心目标：

1. 文件上传后快速返回，不再让页面等待完整解析、切片、Embedding 和质量评分。
2. 普通知识库资料和销售训练资料共用一套入库任务机制。
3. 销售训练资料继续保留“临时切片预览、质量评分、人工发布、正式训练检索”的业务流程。
4. 删除时继续保持 MySQL、MinIO、Qdrant、Redis 全链路一致。

## 二、整体流程

```text
上传文件
  -> MinIO 保存原始文件
  -> MySQL 写 documents
  -> 销售资料额外写 sales_training_knowledge_batches
  -> MySQL 写 ingest_tasks
  -> FastAPI 内部后台 worker 处理
  -> 更新任务进度
  -> 写入 Qdrant 临时库或正式库
  -> 更新 documents / batch 状态
```

## 三、资料类型差异

| 类型 | 入口 | Qdrant 写入目标 | 是否需要人工发布 | 是否需要质量评分 |
|---|---|---|---|---|
| 普通知识库资料 | 首页知识库上传 | 正式知识库 collection | 不需要 | 不需要 |
| 销售训练资料 | 销售训练资料管理上传 | 先写临时 collection | 需要 | 需要 |

销售训练资料不能上传后直接进入正式训练库，必须先进入待发布状态。

## 四、销售训练异步流程

```text
销售训练资料上传
  -> 保存到 MinIO
  -> 创建 documents
  -> 创建 sales_training_knowledge_batches
  -> 创建 ingest_tasks，状态 queued
  -> 页面立即显示一条资料记录
  -> 后台任务开始处理
  -> 解析文件
  -> 规则切片
  -> 必要时 LLM 兜底切片
  -> 生成 quality_report
  -> 写入 sales_training_cases_staging
  -> batch 状态改为 pending_review
  -> 人工查看切片和质量评分
  -> 人工点击发布
  -> 从 staging 复制到 sales_training_cases
  -> batch 状态改为 published
```

## 五、页面入口

异步任务状态不单独做隐藏后台页。

销售训练资料相关状态统一在：

```text
销售陪练 -> 资料管理
```

中展示。

首页知识库页面可以展示文件基础信息，但销售训练专属操作只放在销售训练资料管理页。

销售训练专属操作包括：

1. 查看切片
2. 查看质量评分
3. 发布
4. 重新切分
5. 删除

## 六、销售资料列表展示字段

销售训练资料管理列表建议展示：

| 字段 | 含义 |
|---|---|
| source_file | 文件名 |
| source_type | 来源类型 |
| status | 批次状态 |
| task_status | 入库任务状态 |
| progress | 入库进度 |
| current_step | 当前处理步骤 |
| chunk_count | 切片数量 |
| point_count | 向量点数量 |
| quality_score | 切片质量评分 |
| error_message | 失败原因 |
| created_by | 上传人 |
| updated_at | 更新时间 |

## 七、状态与按钮规则

| batch 状态 | task 状态 | 页面文案 | 查看切片 | 查看评分 | 发布 | 重新切分 | 删除 |
|---|---|---|---|---|---|---|---|
| parsing | queued | 排队中 | 禁用 | 禁用 | 禁用 | 禁用 | 可用 |
| parsing | running | 入库中 | 禁用 | 禁用 | 禁用 | 禁用 | 禁用 |
| parsing | chunking | 正在切片 | 禁用 | 禁用 | 禁用 | 禁用 | 禁用 |
| parsing | llm_chunking | LLM 正在切片 | 禁用 | 禁用 | 禁用 | 禁用 | 禁用 |
| parsing | scoring | 正在生成质量评分 | 禁用 | 禁用 | 禁用 | 禁用 | 禁用 |
| pending_review | succeeded | 待发布 | 可用 | 可用 | 可用 | 可用 | 可用 |
| published | succeeded | 已发布 | 可用 | 可用 | 禁用 | 可用 | 可用 |
| parsing_failed | failed | 入库失败 | 禁用 | 禁用 | 禁用 | 可用 | 可用 |
| delete_failed | 任意 | 删除失败 | 视情况 | 视情况 | 禁用 | 禁用 | 可用 |

第一版可以禁止删除 running 中的任务，避免后台线程正在写 Qdrant 时被并发删除。

后续如果要支持运行中取消，可以增加 `cancel_requested` 状态。

## 八、进度条规则

前端进度条根据 `current_step` 显示。

建议默认进度：

| current_step | progress | 页面文案 |
|---|---:|---|
| queued | 5 | 等待处理 |
| saving | 10 | 文件已保存 |
| parsing | 25 | 正在解析 |
| chunking | 45 | 正在切片 |
| llm_chunking | 60 | LLM 正在优化切片 |
| scoring | 75 | 正在生成质量评分 |
| indexing_staging | 90 | 正在写入临时向量库 |
| succeeded | 100 | 待发布 |
| failed | 100 | 入库失败 |

如果后续能统计总页数或总切片数，可以把进度从固定映射升级为真实进度。

## 九、切片预览兼容

切片预览继续从临时向量库读取：

```text
sales_training_cases_staging
```

读取条件：

```text
batch_id = 当前批次
```

只有当状态为 `pending_review` 或 `published` 时允许查看切片。

`published` 状态查看切片时，优先展示正式库中的切片；如果正式库读取失败，可以回退展示 staging 中的切片并给出提示。

## 十、质量评分兼容

销售训练上传时的质量评分不是训练会话评分。

两者区别：

| 类型 | 生成时机 | 存储位置 | 用途 |
|---|---|---|---|
| 切片质量评分 | 上传解析后 | batch.quality_report | 发布前判断切片质量 |
| 训练会话评分 | 学员完成对话后 | sales_training_scores | 评价学员销售表现 |

异步化后，切片质量评分由后台任务生成。

页面在 `scoring` 状态显示“正在生成质量评分”。

状态进入 `pending_review` 后，前端可以展示完整 `quality_report`。

## 十一、发布兼容

发布操作不异步化，仍然由用户手动触发。

发布前必须满足：

```text
batch.status = pending_review
ingest_task.status = succeeded
staging collection 中存在 batch_id 对应切片
```

发布动作：

```text
删除正式库中同版本组旧 current 批次
从 staging 复制当前 batch_id 切片到正式 collection
更新 metadata：is_current=true、status=published
更新 batch.status=published、is_current=true
同版本组旧批次 is_current=false
```

## 十二、重新切分兼容

重新切分也改为异步任务。

流程：

```text
用户点击重新切分
  -> 删除 staging 中当前 batch_id 旧切片
  -> batch.status 改为 parsing
  -> 创建 ingest_tasks
  -> 后台重新解析、切片、评分、写 staging
  -> 成功后回到 pending_review
```

如果批次已经发布，重新切分不直接影响正式训练库。

只有用户再次点击发布后，新的切片才替换正式库。

## 十三、删除兼容

删除需要覆盖所有状态。

删除资源范围：

```text
documents
sales_training_knowledge_batches
ingest_tasks
MinIO 原文件
Qdrant staging points
Qdrant published points
Redis 临时状态
```

第一版规则：

1. `running/chunking/llm_chunking/scoring/indexing_staging` 状态禁止删除。
2. `queued/failed/pending_review/published` 状态允许删除。
3. 删除失败时保留 MySQL 记录，状态标记为 `delete_failed`。

## 十四、接口交互建议

### 上传销售训练资料

```text
POST /api/v2/training/knowledge/upload
```

响应新增：

```json
{
  "batch_id": "batch_xxx",
  "document_id": "doc_xxx",
  "task_id": "task_xxx",
  "status": "parsing",
  "task_status": "queued",
  "progress": 5,
  "current_step": "queued"
}
```

### 查询销售资料列表

```text
GET /api/v2/training/knowledge/batches
```

列表项补充：

```json
{
  "task_id": "task_xxx",
  "task_status": "running",
  "progress": 45,
  "current_step": "chunking",
  "error_message": null
}
```

### 查询任务状态

```text
GET /api/v2/ingest-tasks/{task_id}
```

用于前端轮询单个任务。

### 重新触发入库

```text
POST /api/v2/ingest-tasks/{task_id}/retry
```

用于失败后重试。

### 重新切分销售资料

```text
POST /api/v2/training/knowledge/batches/{batch_id}/reparse
```

改为创建异步任务，返回 `task_id`。

## 十五、错误处理

后台任务失败时：

```text
ingest_tasks.status = failed
ingest_tasks.error_message = 错误原因
documents.status = failed
sales_training_knowledge_batches.status = parsing_failed
```

前端显示：

```text
入库失败：错误原因
```

可操作：

1. 重新入库
2. 删除

## 十六、服务重启恢复

FastAPI 重启时会执行一次异步入库任务恢复：

```text
查询 ingest_tasks 中 status=running 的任务
  -> 标记为 failed
  -> error_message 写入“服务重启导致任务中断，请手动重试”

查询 ingest_tasks 中 status=queued 的任务
  -> 重新提交到 FastAPI 内部线程池
```

这样处理的原因：

1. `running` 任务对应的是旧进程里的线程，服务重启后线程已经不存在，不能继续假装运行中。
2. `queued` 任务还没有真正开始处理，可以安全重新提交。
3. 失败任务保留在 MySQL 中，页面可以展示失败原因，并通过 `/api/v2/ingest-tasks/{task_id}/retry` 手动重试。

## 十七、设计边界

本次只做项目内部异步入库，不引入 XXL-JOB。

不在第一版支持：

1. 多 worker 分布式消费。
2. 运行中取消任务。
3. 真实百分比进度。
4. 任务优先级队列。

这些能力可以在内部异步稳定后再升级。
