# 合格工程架构实施计划

## 一、实施原则

本计划按低风险到高风险推进，避免一次性大改导致功能不可用。

原则：

1. 每阶段只解决一类问题。
2. 每阶段都能独立提交和回滚。
3. 先不改变业务行为，再逐步优化内部结构。
4. 每阶段必须跑编译检查和核心测试。
5. 涉及删除、入库、模型调用的改动必须补日志。

## 二、阶段一：目录收敛

目标：清理 `app_v2` 历史命名，把业务目录收敛为 `app`。

改造动作：

1. 将 `app_v2/` 重命名为 `app/`。
2. 全局替换 Python import：

```text
from app_v2.xxx import yyy
```

改成：

```text
from app.xxx import yyy
```

3. 检查以下文件是否引用 `app_v2`：

```text
api/main.py
Dockerfile
docker-compose.yml
tests/
docs/
README
```

4. 保持 `api/` 作为启动入口不变。
5. 不改业务逻辑。

验证：

```powershell
.\.venv\Scripts\python.exe -m compileall -q api app core tests
.\.venv\Scripts\python.exe -m pytest tests -q
rg -n "app_v2" .
```

完成标准：

1. 项目中不再有有效 `app_v2` import。
2. FastAPI 能正常启动。
3. 核心接口能正常访问。

## 三、阶段二：销售训练 Service 拆分

目标：降低销售训练核心文件复杂度。

拆分目标：

```text
app/application/training/
  training_knowledge_service.py
  training_role_service.py
  training_session_service.py
  training_score_service.py
  training_query_service.py
```

建议顺序：

1. 先抽 `training_query_service.py`，因为它主要封装向量检索，边界清楚。
2. 再抽 `training_score_service.py`，因为评分逻辑相对独立。
3. 再抽 `training_role_service.py`，处理画像、补充问题、角色生成。
4. 最后抽 `training_knowledge_service.py` 和 `training_session_service.py`。

验证：

```powershell
.\.venv\Scripts\python.exe -m compileall -q app core tests
.\.venv\Scripts\python.exe -m pytest tests -q
```

手工验证：

1. 销售训练资料上传。
2. 生成补充问题。
3. 生成 AI 客户。
4. 开始训练。
5. 流式对话。
6. 结束评分。

## 四、阶段三：考试 Service 拆分

目标：把考试题源、会话、命题、阅卷拆开。

目标目录：

```text
app/application/exam/
  exam_source_service.py
  exam_session_service.py
  exam_question_service.py
  exam_grading_service.py
```

拆分顺序：

1. `exam_source_service.py`：Qdrant 题源查询和章节统计。
2. `exam_question_service.py`：题型转换、模型命题、规则兜底。
3. `exam_grading_service.py`：模型阅卷、规则评分。
4. `exam_session_service.py`：会话创建、历史、删除、详情。

完成标准：

1. 路由层只做请求参数和响应模型转换。
2. 考试 Service 之间依赖方向清楚。
3. 原有考试接口响应结构不变。

## 五、阶段四：RAG Pipeline 抽象

目标：把知识库问答链路整理成可测试管道。

目标目录：

```text
core/rag/pipeline/
  query_planner.py
  retriever.py
  reranker.py
  context_builder.py
  answer_generator.py
  evaluation_logger.py
```

执行顺序：

1. 保留现有 `KnowledgeAnswerService` 对外接口。
2. 内部逐步改成调用 pipeline 组件。
3. 先接入现有 Query Planner 和 Qdrant 检索。
4. 再抽 Context Builder。
5. 最后预留 Reranker 接口。

完成标准：

1. `/chat` 和 `/chat/stream` 行为不变。
2. 日志能看到每个阶段耗时。
3. 单元测试能分别测试 query planner、context builder。

## 六、阶段五：RAG 评估集

目标：让 RAG 效果可量化。

新增文件：

```text
tests/rag_eval/
  rag_eval_cases.yml
  test_rag_retrieval_eval.py
```

首批建议准备 30 条问题：

1. 编号问答类。
2. 同义改写类。
3. 多问题类。
4. 资料不足类。
5. 销售训练案例类。

评估字段：

```yaml
- case_id: robot_001
  question: 中国市场啥时候开始普及扫地机器人？
  expected_keywords:
    - 2015
    - 小米生态链
  expected_sources:
    - 扫地机器人100问.pdf
```

完成标准：

1. 能输出召回命中率。
2. 能输出关键词命中率。
3. 能记录失败 case，便于后续调参。

## 七、阶段六：异步入库任务

> 当前状态：暂缓。本阶段按用户要求先放放，不在本轮实现。

目标：上传文件后快速返回任务编号，后台完成解析和入库。

建议先用轻量方案：

```text
FastAPI BackgroundTasks / ThreadPoolExecutor
```

后续再升级：

```text
Celery / Dramatiq / ARQ
```

需要新增任务表或扩展现有 documents 状态：

```text
task_id
document_id
status
current_step
progress
error_message
created_at
updated_at
```

完成标准：

1. 上传接口不再等待完整 embedding。
2. 前端可以轮询进度。
3. 失败可以查看错误原因。
4. 可以重新触发入库。

## 八、阶段七：全链路删除补偿

> 当前状态：已完成本轮增强。删除入口保留原有 `status` 和业务 ID 字段，同时返回 `resource_results`、`errors`，用于查看 Qdrant、MinIO、Redis、MySQL 每类资源处理结果。

目标：删除资料时 MySQL、Qdrant、MinIO、Redis 一致。

删除流程：

```text
查 MySQL
 -> 删除 Qdrant points
 -> 删除 MinIO object
 -> 删除 Redis 状态
 -> 删除 MySQL 记录
 -> 记录删除结果
```

失败处理：

1. Qdrant 删除失败：记录错误，保留 MySQL 记录状态为 `delete_failed`。
2. MinIO 删除失败：记录对象名，允许补偿删除。
3. MySQL 删除失败：不影响已删除外部资源，但要记录补偿日志。

完成标准：

1. 删除接口能返回每类资源删除结果。
2. 日志里能用 document_id 串起完整删除链路。
3. 重复删除不报 500。

## 九、阶段八：可观测性增强

> 当前状态：已完成本轮增强。保留原有 `/api/v2/health`，新增 `/api/v2/health/dependencies` 返回 MySQL、Redis、MinIO、Qdrant、聊天模型配置、Embedding 模型配置的健康明细；聊天链路补充接口层 `trace_id` 并向 SSE、RAG 日志和会话 metadata 透传；RAG 检索补充 `query_count/retrieved_count/reranked_count/collection_name/total_ms` 汇总日志；知识库入库补充 `document_id/collection_name/chunk_count` 汇总日志。

目标：定位慢点、失败点、召回质量问题。

新增日志字段：

```text
trace_id
conversation_id
document_id
collection_name
model_name
query_count
retrieved_count
reranked_count
first_token_ms
total_ms
```

建议新增健康检查：

```text
GET /health
GET /health/dependencies
```

依赖检查：

1. MySQL。
2. Redis。
3. MinIO。
4. Qdrant。
5. Embedding 模型可选。
6. Chat 模型可选。

## 十、风险与回滚

| 风险 | 阶段 | 处理方式 |
|---|---|---|
| import 替换遗漏 | 阶段一 | compileall + rg `app_v2` |
| 拆 Service 后行为变了 | 阶段二/三 | 保持路由响应不变，补核心接口测试 |
| RAG pipeline 抽象过度 | 阶段四 | 先包现有逻辑，不新增复杂接口 |
| 评估集太理想化 | 阶段五 | 使用真实问法和失败 case |
| 异步任务状态不一致 | 阶段六 | 增加任务状态和错误日志 |
| 删除链路半失败 | 阶段七 | 先外部资源后 MySQL，记录补偿 |

## 十一、建议提交节奏

建议每个阶段一个分支或一个清晰 commit：

```text
1. 架构目录收敛
2. 销售训练服务拆分
3. 考试服务拆分
4. RAG pipeline 抽象
5. RAG 评估集
6. 异步入库任务
7. 全链路删除补偿
8. 可观测性增强
```

这样每一阶段都能单独评审、回滚和验证。
